# 第 3 章：主流模型与能力边界

## 本章解决什么问题

前两章解决了两个基础问题：AI、机器学习、深度学习、生成式 AI、LLM 和 Agent 之间是什么关系；LLM 为什么能生成，也为什么会出错。本章要解决第三个问题：真实做 Agent 系统时，应该怎么选模型？

很多初学者会把模型选择简化成一句话：

> 选最强的模型。

这在 Demo 阶段可能能跑通，但在真实系统里很快会出问题。因为模型不是越强越好，而是要和任务、成本、延迟、上下文、工具调用、部署方式、安全要求和评估方式匹配。

本章要回答：

- Chat 模型、推理模型、多模态模型、Embedding 模型、代码模型分别适合做什么？
- 为什么同一个模型不适合所有 Agent 任务？
- 主流模型家族应该如何理解，而不是如何背版本号？
- 如何在效果、成本、延迟、上下文、工具调用能力之间做取舍？
- Java 后端工程师如何把模型选择落到模型网关、配置、评估和降级策略里？

截至 2026-05，主流模型迭代非常快，具体版本、价格、上下文长度和可用区域都可能变化。因此本章不把“某个版本一定最好”写成结论，而是建立一套可复用的模型选择方法。

## 一个直观例子

继续使用会议纪要助手。这个系统看起来只是“把会议内容总结出来”，但如果拆开任务，会发现它不应该只用一个模型：

| 任务 | 适合的模型类型 | 为什么 |
| --- | --- | --- |
| 语音转文字 | 音频/转写模型 | 输入是音频，不是普通文本 |
| 查找历史项目资料 | Embedding 模型 + 检索系统 | 需要语义相似度，而不是直接生成 |
| 总结会议内容 | 通用 Chat 模型或较强语言模型 | 需要理解和压缩文本 |
| 抽取行动项 JSON | 支持结构化输出的语言模型 | 输出要被程序消费 |
| 判断风险和冲突 | 推理能力更强的模型 | 需要跨信息综合判断 |
| 生成给用户看的报告 | 语言表达较好的模型 | 需要可读性和稳定风格 |
| 写入任务系统前校验 | 后端规则 + 权限系统 | 不能交给概率模型决定 |

如果你把所有步骤都交给一个最贵、最强的模型，系统可能能跑，但成本高、延迟长，也不一定更可靠。如果你为了省钱全部用小模型，摘要可能可以，风险判断和复杂抽取可能明显下降。

专业的模型选择不是追逐名字，而是把任务拆开，再给每类任务选择合适的模型和工程保护。

后文会沿着这个例子继续推进：先区分模型类型，再看能力维度，然后把选择落到模型路由和模型网关，最后用评估集验证选择是否真的有效。

## 基础解释

### Chat 模型：通用语言交互入口

Chat 模型主要用于自然语言对话、问答、总结、改写、解释、抽取和轻量规划。它们通常是 Agent 系统的通用入口，因为用户大多数时候用自然语言表达目标。

适合 Chat 模型的任务包括：

- 回答用户问题。
- 总结文档。
- 改写文本。
- 提取简单字段。
- 生成用户可读说明。
- 根据上下文给出下一步建议。

但 Chat 模型不应该直接承担所有责任。权限判断、金额计算、事务写入、合规审批、生产配置修改，都应该由确定性系统处理。

在会议纪要助手里，Chat 模型适合生成摘要和解释行动项，但不适合直接决定是否把任务写入项目管理系统。

### 推理模型：给复杂任务更多思考预算

推理模型通常针对复杂问题、多步推理、代码、数学、规划、工具密集任务做了优化。它们的特点不是简单“更聪明”，而是更适合处理需要分解、检查、权衡和长期上下文跟踪的任务。

本章里的“推理模型”是工程分类，不是所有厂商统一命名的严格类别。它通常指针对复杂推理、可调思考预算或长链路任务优化的模型或模式。

适合推理模型的任务包括：

- 多步骤问题分析。
- 复杂代码修改和调试。
- 需要审查多个约束的计划生成。
- 工具调用前的风险判断。
- 多文档信息综合。
- 需要跨约束检查的结构化分析。

推理模型的代价通常是更高成本和更长延迟。工程上不能因为任务“看起来重要”就全部使用推理模型，而应该判断这个步骤是否真的需要深度推理。推理能力也不等于结构化输出保证，JSON、工具参数和业务字段仍然需要 Schema 约束、解析校验和失败处理。

在会议纪要助手里，普通摘要可以用 Chat 模型；但如果要判断“这次会议的上线计划是否和历史风险冲突”，就更适合交给推理能力更强的模型。

### 多模态模型：处理文本之外的输入

多模态模型可以处理文本之外的一种或多种输入，例如图片、音频、视频、文档截图等。它们对于 Agent 很重要，因为真实工作并不总是干净文本：用户可能上传截图、合同 PDF、白板照片、录音、网页截图或表格图片。

适合多模态模型的任务包括：

- 读图和图表理解。
- 截图问答。
- 文档版面理解。
- 音频对话或实时语音交互。
- 图片内容描述。
- 多模态报告生成。

多模态能力也有边界。模型能“看见”图片，不等于能可靠识别所有细节；能处理音频，不等于能替代专业转写和质检流程；能读文档截图，不等于能保证法律或财务信息绝对准确。

工程上要把多模态模型的结果当作可用输入，而不是最终事实。重要信息仍然需要 OCR、结构化解析、人工复核或业务系统校验。

具体能力必须按模型文档确认，尤其要区分输入模态、输出模态、实时能力、文件或 PDF 处理能力、是否支持工具调用。不要因为一个模型标注“多模态”，就假设它同时适合图片理解、实时语音、视频分析和文档抽取。

### Embedding 模型：检索和相似度的基础能力

Embedding 模型通常把文本编码成向量，用于相似度计算、聚类、去重、推荐和 RAG 检索。部分模型或平台支持图像、多模态向量，但需要单独确认模型能力、向量空间是否兼容和检索评估方式。

它和 Chat 模型的用途不同：

- Chat 模型负责生成。
- Embedding 模型负责表示和检索。

在 RAG 系统中，Embedding 模型决定“哪些资料可能和用户问题相关”。如果召回错了，后面的生成模型再强，也可能基于错误材料回答。

选择 Embedding 模型时，要关注：

- 支持的语言。
- 对领域术语的表现。
- 向量维度和存储成本。
- 检索效果。
- 延迟和吞吐。
- 是否支持多模态向量。

在会议纪要助手里，Embedding 模型可以帮助找出历史会议、项目文档、风险记录和当前会议之间的语义关联。

### 代码模型和 Agentic Coding 模型

代码模型或面向编码任务优化的模型，更适合代码阅读、修改、测试、解释、重构和长任务工程协作。它们通常需要处理大量文件、工具调用、终端输出和错误日志。

适合代码模型的任务包括：

- 阅读代码库。
- 定位 Bug。
- 生成补丁。
- 编写测试。
- 分析构建失败。
- 长时间多步骤开发任务。

但代码模型也不能绕过工程流程。任何自动生成的代码都要经过测试、Review、安全扫描和版本控制。Agent 写代码不是替代工程纪律，而是把工程纪律自动化、半自动化。

## 核心原理

### 模型选择的本质：任务和能力匹配

模型选择不是品牌选择，而是任务和能力匹配。一个实用的模型选择流程可以这样理解：

```text
任务类型
  -> 输入模态
  -> 输出形式
  -> 推理深度
  -> 工具需求
  -> 风险等级
  -> 延迟要求
  -> 成本预算
  -> 部署与合规约束
  -> 选择模型与兜底策略
```

这条链路比“哪个模型最强”更重要。

例如，一个客服 FAQ 问答系统，真正关键的可能是检索准确率、引用来源和拒答边界，不一定需要最强推理模型。一个代码开发 Agent，真正关键的可能是长上下文、工具调用、代码理解和测试反馈。一个财务审批 Agent，真正关键的可能是权限、审计、人工确认和确定性规则，而不是模型自由发挥。

### 能力维度：不要只看一个分数

评价模型时，至少要拆成多个维度：

| 维度 | 要问的问题 |
| --- | --- |
| 语言理解 | 能否正确理解用户目标、业务术语和上下文？ |
| 推理能力 | 能否处理多步骤约束、冲突和复杂判断？ |
| 指令遵循 | 能否稳定遵循 System Prompt、格式和边界？ |
| 结构化输出 | 能否稳定输出 JSON、字段、枚举和工具参数？ |
| 工具调用 | 能否生成可校验的工具调用意图和参数？ |
| 多模态 | 是否需要图片、音频、视频或文档截图输入？ |
| 长上下文 | 是否需要处理长文档、长对话或代码库？ |
| 成本 | 单次任务成本和重试成本是否可接受？ |
| 延迟 | 用户是否能接受等待？是否需要流式输出？ |
| 可部署性 | 是否能满足私有化、区域、合规和数据边界？ |

不同项目的权重不同。个人笔记助手可能更看重成本和速度；企业合同审查更看重引用、权限、审计和人工确认；代码 Agent 更看重代码能力、工具调用和长任务稳定性。

以会议纪要助手为例，可以这样做一次完整取舍：

| 子任务 | 推荐选择 | 取舍理由 | 兜底方式 |
| --- | --- | --- | --- |
| 普通会议摘要 | 快速低成本 Chat 模型 | 语言压缩任务，不一定需要深度推理 | 保留原文引用，允许用户编辑 |
| 行动项 JSON 抽取 | 支持结构化输出的 Chat/推理模型 | 输出要进入程序，需要格式稳定 | Schema 校验、字段修复、人工确认 |
| 历史资料召回 | 文本 Embedding + 检索 | 先找相关资料，再让生成模型阅读 | 权限过滤、Rerank、引用来源 |
| 上线风险判断 | 推理能力更强的模型 | 需要综合会议内容和历史风险 | 给出依据，风险高时转人工 |
| 写入任务系统 | 后端规则和工具服务 | 这是业务副作用，不是生成任务 | 权限、幂等、审批和审计 |

这张表不是固定答案，而是示范如何把“模型能力”翻译成“系统责任”。

### 模型生态：按选型维度看，而不是按品牌背

截至 2026-05，主流模型生态变化很快。具体模型名称、版本、价格、上下文长度、区域可用性和废弃计划必须以官方文档为准。不要把今天的模型表当成长期架构设计的一部分。

工程选型时，更有用的不是记住供应商清单，而是按维度判断：

| 选型维度 | 要确认的问题 |
| --- | --- |
| 托管 API | 是否需要快速接入、托管运维和供应商最新能力？数据能否发送到外部服务？ |
| 开放权重 | 是否需要私有化部署、可控数据边界、定制优化或离线环境？团队是否有推理运维能力？ |
| 中文和本地生态 | 是否需要稳定处理中文业务术语、本地知识库、中文代码注释和中文用户表达？ |
| 多模态能力 | 是否真的需要图片、音频、视频、PDF 或截图输入？对应模型是否支持目标输入和输出形式？ |
| 长上下文 | 是否需要处理长文档、长对话、代码库或大量检索材料？长上下文效果是否经过业务评估？ |
| 工具调用和结构化输出 | 是否需要稳定生成工具参数、JSON、枚举和业务字段？是否有 Schema 校验和修复链路？ |
| 推理与代码能力 | 是否需要多步骤推理、复杂代码修改、测试反馈和长任务规划？是否能接受更高成本和延迟？ |
| 区域和合规 | 供应商、部署区域、日志留存、数据处理方式是否满足安全和合规要求？ |

不同供应商和开放模型生态会在这些维度上有不同组合。选型时可以先用这张表筛掉不满足硬约束的候选，再用业务评估集比较效果、成本、延迟和失败类型。

### 闭源模型、开放模型和自部署

模型选择还涉及部署方式。

闭源或托管 API 模型的优势通常是接入快、托管运维少、可快速使用供应商最新能力。具体能力强弱要按任务评估。代价是供应商依赖、数据边界、成本波动、区域可用性和接口变更风险。

开放权重模型的优势是可私有化、可微调、可控数据边界、可在特定硬件上优化。代价是部署、推理优化、模型评估、安全治理和运维成本更高。

自部署不是“免费”。你省下的是一部分 API 调用费用，但会增加 GPU、推理服务、监控、扩缩容、安全、升级和模型评估成本。

企业系统常见做法不是二选一，而是混合使用：普通任务用自部署或低成本模型，复杂任务路由到高能力模型，高敏感数据留在私有环境，公开知识或低风险任务使用云模型。

## 工程实现

### 从直接调用模型到模型网关

Demo 阶段可以在业务代码里直接调用模型 API。生产系统不建议这样做。更稳的方式是引入模型网关或模型服务层。

模型网关负责：

- 屏蔽不同模型供应商的接口差异。
- 统一认证、限流、超时和重试。
- 记录模型、Prompt、参数和版本。
- 做模型路由和降级。
- 控制成本预算。
- 统一日志、指标和 Trace。
- 支持灰度发布和回滚。

一个简化结构如下：

```text
业务服务
  -> Agent Runtime
  -> Model Gateway
     -> OpenAI / Claude / Gemini / Qwen / DeepSeek / 自部署模型
  -> Observability & Eval
```

这样做的好处是：业务代码不需要到处写供应商判断；模型升级也不会变成全系统搜索替换。

模型网关的后端边界可以更具体一些：

- 统一请求协议：把不同业务请求转换成统一的 model_request。
- 供应商 Adapter：封装 OpenAI、Claude、Gemini、Qwen、DeepSeek、自部署模型等接口差异。
- 路由器：根据任务、风险、租户、数据边界、预算和可用性选择模型。
- 策略和配置中心：管理默认模型、候选模型、降级模型、灰度比例和上线门禁。
- 调用执行器：处理超时、重试、熔断、并发和流式响应。
- 日志审计：记录请求、模型、参数、检索材料、工具调用和最终业务动作。
- 评估回放入口：把失败样本和灰度样本送入评估集。

这些模块不一定都要独立成服务，但职责要清楚。否则模型网关会变成一个到处堆 if else 的代理层。

### 模型路由：按任务选择模型

模型路由可以先从简单规则开始，不一定一开始就做复杂智能调度：

| 任务类型 | 路由策略 |
| --- | --- |
| 简单摘要、改写 | 快速低成本 Chat 模型 |
| 复杂推理、风险判断 | 推理模型或更高能力模型 |
| 图片、截图、音频 | 多模态模型或专用模型 |
| 语义检索 | Embedding 模型 |
| 代码修改 | 代码模型或擅长代码的推理模型 |
| 高风险写入 | 模型只生成候选，后端校验和人工确认 |

后续可以引入更细的策略，例如根据用户等级、任务风险、预算、上下文长度、实时性要求和历史效果动态选择模型。

一个简化路由决策顺序可以是：

1. 检查租户、区域和数据出域规则，不合规则直接拒绝或选择私有模型。
2. 检查任务风险和权限，高风险任务只允许进入带人工确认的模型链路。
3. 检查输入模态和上下文长度，选择支持对应能力的候选模型。
4. 检查预算、SLA 和队列状态，决定默认模型、快速模型或异步执行。
5. 检查模型可用性和灰度策略，选择 requested_model 和 fallback_model。
6. 记录选择原因，方便后续排障和评估。

会议纪要助手可以先用低成本 Chat 模型做摘要；如果风险判断需要跨历史资料推理，再路由到推理模型；如果模型服务限流，摘要可以降级，风险判断则宁可转人工，也不要自动换成能力不足的模型。

### 输入、输出、状态和异常

模型选择要落到真实接口，就必须定义数据契约。

输入要包含：task_type、risk_level、input_modality、expected_output、context_size、user_id、tenant_id、trace_id、budget 和 deadline。

处理要包含：模型选择规则、Prompt 版本、采样参数、工具可用范围、超时、重试和降级策略。

输出要包含：requested_model、resolved_model、model_snapshot 或 deployment_id、output、usage、latency、finish_reason、safety_result、structured_validation_result 和 fallback_action。不同供应商对版本字段的口径不同，生产系统要按供应商能力做映射。

状态要包含：任务选择了哪个模型、为什么选择、是否降级、是否重试、是否命中缓存、是否触发人工确认。

一个会议纪要助手的模型网关请求可以抽象成下面这样。这里是伪 JSON，用来说明字段关系，不代表某个供应商的真实 API：

```json
{
  "trace_id": "trace_20260526_001",
  "tenant_id": "team-a",
  "user_id": "u-123",
  "task_type": "meeting_summary",
  "risk_level": "low",
  "input_modality": ["text"],
  "expected_output": "markdown_summary",
  "context_size": 18000,
  "budget": {
    "max_cost_cents": 20,
    "max_output_tokens": 1200
  },
  "deadline_ms": 8000,
  "routing_policy": {
    "allowed_models": ["fast-chat", "balanced-chat"],
    "default_model": "fast-chat",
    "fallback_model": "balanced-chat",
    "fallback_policy": "same_risk_level_only"
  },
  "validation": {
    "schema": "meeting-summary-v1",
    "requires_human_approval": false
  }
}
```

模型网关返回时，业务服务至少要拿到可复盘的信息：

```json
{
  "trace_id": "trace_20260526_001",
  "requested_model": "fast-chat",
  "resolved_model": "fast-chat",
  "model_snapshot": "provider-managed-snapshot",
  "prompt_version": "meeting-summary-v3",
  "routing_reason": "low_risk_summary_under_budget",
  "usage": {
    "input_tokens": 4200,
    "output_tokens": 860
  },
  "latency_ms": 3100,
  "finish_reason": "stop",
  "structured_validation_result": "passed",
  "fallback_action": "none"
}
```

真实系统还要根据敏感级别决定是否保存原始 Prompt、用户输入、检索片段和模型输出。可观测性不是把所有内容无差别落库，而是让排障、评估和审计拿到足够信息。

异常至少要分成：

- 模型不可用。
- 供应商限流。
- 上下文超限。
- 成本超预算。
- 输出格式不合格。
- 安全策略拦截。
- 工具参数校验失败。

不同异常要有不同处理。比如上下文超限可以摘要或检索；成本超预算可以降级或请求确认；安全拦截应该终止并记录；输出格式不合格可以修复或重试。

可以用一个最小矩阵约束降级边界：

| 异常类型 | 自动重试 | 自动降级 | 转人工 | 告警/评估 |
| --- | --- | --- | --- | --- |
| 模型超时或临时不可用 | 可以 | 可以，限低风险任务 | 视任务而定 | 记录 |
| 供应商限流 | 可以退避重试 | 可以切换同等级模型 | 长任务可转异步 | 记录 |
| 上下文超限 | 不适合直接重试 | 可以摘要、检索或裁剪 | 需要用户补充时转人工 | 记录样本 |
| 输出格式不合格 | 可以有限重试 | 可换同等级模型 | 多次失败转人工 | 进入评估集 |
| 权限失败 | 不应重试 | 不应降级绕过 | 必要时请求授权 | 必须审计 |
| 安全策略拦截 | 不应重试 | 不应降级绕过 | 高风险转人工 | 必须告警 |
| 高风险写入不确定 | 不应自动执行 | 不应换弱模型执行 | 必须确认 | 必须审计 |

### 评估驱动模型选择

不要凭感觉换模型。模型选择应该由评估集驱动。

一个最小评估集可以包括：

- 典型用户问题。
- 长文档问题。
- 多轮对话问题。
- 工具调用参数生成。
- 结构化输出。
- 权限边界测试。
- 幻觉诱导问题。
- 多语言和领域术语。
- 失败案例回放。

评估指标可以包括：

- 任务成功率。
- 格式通过率。
- 引用准确性。
- 工具参数合法率。
- 人工接管率。
- 平均延迟和 P95 延迟。
- 单任务成本。
- 用户反馈。
- 高风险操作拦截率。

模型升级前，至少要用同一批评估样本比较旧模型和新模型。否则模型升级可能带来隐藏回归：摘要更流畅了，但 JSON 更不稳定；推理更强了，但成本翻倍；工具参数更积极了，但误调用增加。

评估还应该进入发布流程，而不是停留在报告里。例如：离线评估没有达到最低格式通过率，不允许灰度；灰度阶段 P95 延迟、单任务成本或人工接管率超过阈值，自动回滚；高风险误调用出现一次，就暂停扩大流量并进入人工复盘。具体阈值由业务设定，本章不写死数字。

## 适用场景

### 玩具 Demo

Demo 阶段可以选择一个通用 Chat 模型，先跑通对话、摘要和简单结构化输出。这个阶段重点是理解模型差异，不需要搭复杂模型网关。

但 Demo 不应承担生产风险。不要因为 Demo 里一个模型表现好，就直接推断它适合所有真实用户和所有业务数据。

### 个人效率工具

个人工具可以优先选择成本可控、速度快、体验好的模型。比如笔记总结、阅读辅助、代码解释、日常问答，不一定每次都用最强模型。

个人工具要关注：

- 每天或每月成本。
- 响应速度。
- 隐私数据是否会发送到外部服务。
- 是否需要离线或本地模型。
- 是否能手动修正输出。

### 团队内部工具

团队工具需要更严格的模型选择。因为多人共用后，错误会变成协作成本，成本会变成团队预算，权限会变成安全问题。

团队内部工具至少要有：

- 统一模型配置。
- Prompt 和模型版本记录。
- 用户和租户维度成本统计。
- 模型调用日志。
- 失败样本回放。
- 模型升级评估。

例如团队知识库问答系统，不仅要选生成模型，还要选 Embedding 模型、Rerank 模型、引用策略和拒答策略。

### 企业级系统

企业系统的模型选择要进入治理体系。模型不是开发者随手填的字符串，而是受安全、合规、预算、SLA 和采购影响的基础设施。

企业级模型治理需要考虑：

- 数据是否允许出域。
- 模型供应商是否满足合规要求。
- 是否需要私有化部署。
- 是否有模型降级和供应商切换方案。
- 是否能审计每次模型调用。
- 是否能限制高风险工具调用。
- 是否有上线前评估和回滚条件。

企业里常见的成熟做法是：模型能力分层、任务风险分级、调用统一走模型网关、所有升级先过评估集。

## 不适用场景

不适合为了“追新”频繁切换模型。模型升级应该解决具体问题，例如降低延迟、提升结构化输出稳定性、支持新的输入模态或满足合规要求，而不是跟着发布节奏漂移。

不适合用单一模型承载所有任务。一个模型可以很强，但 Embedding、转写、图像理解、推理、代码、结构化输出和高风险审批的输入、输出、风险和验证方式不同。

不适合在没有业务评估集时做生产切换。公开排行榜、主观体验和单次试用都只能作为候选筛选，不能替代上线前评估。

不适合把高风险副作用交给模型自由决定。涉及写库、发消息、扣款、审批、权限变更和生产配置修改时，模型最多生成候选动作，最终执行必须由后端规则、权限和人工确认控制。

不适合在数据边界不清楚时调用外部模型。只要涉及客户数据、内部文档、合同、财务、人事或受监管数据，就必须先确认数据分级、区域、留存、脱敏和访问控制。

不适合把模型选择权完全交给模型自己。模型可以帮助判断任务类型，但最终可用模型、工具权限、预算和风险策略应该由后端系统控制。

## 常见坑与反模式

第一个坑是“最强模型万能论”。症状是所有任务都路由到最高能力模型，结果成本和延迟失控。修正方式是按任务拆分模型，并给低风险任务配置更便宜、更快的路径。

第二个坑是“便宜模型省钱论”。症状是单次调用价格下降，但错误率、重试率、人工接管率上升。修正方式是计算任务总成本，而不是只看模型单价。

第三个坑是“榜单第一适合我”。症状是公开 benchmark 很好，业务数据上却引用错、格式错、工具参数错。修正方式是建立自己的评估集，并保留失败样本回放。

第四个坑是“上线后再观察”。症状是模型切换没有门禁，问题只能靠用户反馈发现。修正方式是先离线评估，再灰度，再监控，再扩大范围。

第五个坑是“不记录模型和 Prompt 版本”。症状是同一个请求今天和昨天表现不同，却无法复盘。修正方式是记录 requested_model、resolved_model、model_snapshot 或 deployment_id、prompt_version 和关键参数。

第六个坑是“没有降级边界”。症状是服务不可用时随意换模型，高风险任务被弱模型继续执行。修正方式是定义哪些任务可自动降级，哪些任务必须转人工或终止。

第七个坑是“忽视废弃计划”。症状是供应商下线旧模型或改变可用性时，生产链路突然失败。修正方式是监控官方变更、保留候选模型，并在评估集里提前验证迁移路径。

## 安全、成本与性能考虑

安全方面，模型选择要考虑数据边界。某些数据不能发送到外部 API，某些任务必须使用私有化或指定区域服务。模型输出不能绕过权限系统，模型路由也不能把高敏感数据路由到不合规模型。

权限必须由后端执行。模型只能生成候选工具调用或候选参数，后端要根据用户、租户、资源、动作和风险等级做授权。授权失败不能靠换模型解决，应该终止、记录审计，必要时进入审批流程。

成本方面，要计算任务总成本，而不是单次调用价格。总成本包括输入 Token、输出 Token、工具调用、检索、重试、人工接管、缓存命中率和失败处理。

成本控制要变成配置：预算配额、租户限额、请求级 max tokens、上下文裁剪、缓存策略、请求合并、批处理、排队超时和模型路由。否则成本监控只能告诉你花多了，不能阻止系统继续花。

性能方面，要按用户体验设计模型调用。同步交互更看重首 Token 时间和总延迟；长任务可以异步执行；复杂推理可以用任务状态和通知机制；批量任务要考虑吞吐、队列和限流。

可靠性方面，要准备模型不可用、供应商限流、输出不合格和成本超预算的降级策略。可选降级包括：换模型、缩短上下文、关闭非关键步骤、返回部分结果、转人工确认、延迟执行。

治理方面，要记录每次模型调用的 trace_id、requested_model、resolved_model、model_snapshot 或 deployment_id、prompt_version、task_type、risk_level、input_tokens、output_tokens、latency、cost_estimate、fallback_action 和 validation_result。

日志也要有边界。必须记录的是可复盘字段，例如 trace_id、requested_model、resolved_model、prompt_version、routing_reason、retrieval_doc_ids、tool_call_id、validation_result、fallback_action 和最终业务动作。Prompt、原始输入、检索片段和模型输出是否落库，要按敏感级别脱敏、采样、加密或限制访问，不能为了排障无脑保存全文。

## 如何评估效果

模型选择是否正确，不能靠“感觉这个模型更聪明”。至少要通过三层评估。

第一层是离线评估。用固定样本比较多个模型，包括正常样本、边界样本、失败样本和攻击样本。

第二层是灰度评估。让少量真实流量使用新模型，观察成功率、延迟、成本、用户反馈、人工接管率和错误类型。

第三层是线上持续评估。上线后持续收集失败样本，把它们加入评估集，形成回放机制。

可以用下面的问题自测：

- 是否知道这个任务为什么需要这个模型？
- 是否有比它更便宜或更快的可选模型？
- 是否知道失败时降级到哪里？
- 是否记录了模型版本和 Prompt 版本？
- 是否有业务评估集，而不是只看公开榜单？
- 是否区分了摘要、推理、结构化输出、检索、多模态和工具调用任务？
- 是否知道模型升级会影响哪些系统？

如果这些问题答不上来，说明模型选择还停留在“调用 API”阶段。

## 实践任务

1. 最小任务：为会议纪要助手列出至少 6 个子任务，并为每个子任务选择模型类型。交付物是一张表，包含任务、输入、输出、模型类型、风险和校验方式。

2. 工程化任务：设计一个模型网关配置表，至少包含 task_type、tenant_scope、data_classification、region_constraint、allowed_models、default_model、fallback_model、fallback_policy、max_tokens、timeout、budget、risk_level、retry_policy、rate_limit、validation_schema、eval_gate、rollout_percent、owner、requires_human_approval。验收标准是：能解释每个字段如何影响模型选择。

3. 进阶任务：构建一个小型模型评估集，包含 20 条样本，覆盖摘要、结构化输出、幻觉诱导、权限边界、工具参数生成和长上下文。每条样本至少包含输入、期望行为、评分规则和失败类型。验收标准是：能用同一批样本比较两个模型，并输出效果、成本、延迟和失败类型分析。

参考答案要点：最小任务中，语音转写、Embedding 检索、摘要、行动项结构化抽取、上线风险判断和任务系统写入应被拆成不同子任务；写入任务系统不应交给模型直接决定。模型网关配置必须能表达数据分级、区域约束、降级策略、预算、超时、评估门禁和人工确认。评估集不能只包含正常摘要样本，必须覆盖权限失败、结构化输出失败、无依据回答和高风险动作样本。

## 从入门到专业

- 入门：知道不同模型类型适合不同任务，不再把模型选择理解成“选最强”。
- 初级：能区分 Chat、推理、多模态、Embedding、代码模型的基本用途。
- 中级：能在后端系统中配置模型路由、降级和日志记录。
- 高级：能用业务评估集比较模型，控制成本、延迟和风险。
- 专业：能设计企业级模型治理体系，包括模型网关、评估集、灰度发布、回滚、供应商切换和合规边界。

## 本章小结

模型选择是 Agent 工程的核心能力之一。一个专业系统不会简单地问“哪个模型最强”，而是问：这个任务需要什么输入、什么输出、多少推理、哪些工具、什么风险、多少成本、多少延迟、什么部署边界。

Chat 模型适合通用语言任务，推理模型适合复杂约束和多步骤问题，多模态模型处理文本之外的输入，Embedding 模型支撑检索，代码模型适合工程协作。闭源 API 模型接入快，开放模型和自部署带来更多控制，但也带来运维和评估成本。

对 Agent 开发者来说，模型选择最终应该落到模型网关、模型路由、评估集、日志、降级和治理策略里。模型不是一个写死在代码里的字符串，而是生产系统里需要持续管理的基础设施。

下一章会进入 Prompt Engineering 基础。理解模型能力边界之后，我们才能更清楚地设计 System、User、Assistant 消息、输出格式和约束条件。
