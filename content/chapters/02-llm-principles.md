# 第 2 章：LLM 的基本原理

## 本章解决什么问题

上一章我们建立了 AI、机器学习、深度学习、生成式 AI、LLM 和 Agent 的关系地图。本章要进一步回答一个更底层的问题：大语言模型到底是怎么工作的？

这个问题很重要。因为很多 Agent 工程问题，表面上看是 Prompt 不好、模型不够强、工具调用失败，背后其实都和 LLM 的基本机制有关：

- 为什么模型输入要计算 Token？
- 为什么同一句话换一种写法，模型回答可能变化？
- 为什么模型可以总结、翻译、写代码，却也会一本正经地编造内容？
- 为什么上下文窗口不是“无限记忆”？
- 为什么结构化输出需要校验？
- 为什么模型回答看起来像推理，但不能直接当成数据库事实？

本章不把读者带进完整数学推导，而是建立一个工程师必须掌握的最小模型：

> LLM 把文本切成 Token，把 Token 转成向量表示，通过 Transformer 和 Attention 在上下文中建模关系，最后根据概率分布选择或采样下一个 Token。它的能力来自大规模训练和对齐，但它的输出仍然是概率生成结果，需要工程系统提供上下文、工具、校验和边界。

读完本章，读者应该能够解释 LLM 的基本数据流，也能把这些原理映射到 Agent 系统里的上下文工程、工具调用、结构化输出、RAG、评估和安全问题。

## 一个直观例子

假设你问一个模型：

```text
请把下面这段会议记录总结成 3 条行动项。
```

然后你粘贴了一段会议记录。模型返回：

```text
1. 张三负责整理需求清单。
2. 李四负责确认上线时间。
3. 王五负责补充风险评估。
```

从用户体验看，它像是在“读懂会议记录并思考”。但从模型机制看，事情大致可以拆成几步：

1. 输入文本会被切成 Token。
2. Token 会被转换成向量，也就是模型能处理的数字表示。
3. 模型在上下文中计算每个 Token 和其他 Token 的关系。
4. 模型根据当前上下文计算下一个 Token 的概率分布。
5. 一个 Token 接一个 Token 地生成，直到形成完整回答。

这解释了一个关键事实：模型不是从数据库里查出“正确答案”，也不是像传统程序一样执行固定规则。它是在给定上下文下，根据概率分布逐步选择或采样输出。

这也是为什么同一个会议记录，如果你少粘贴了一段关键信息，模型可能漏掉行动项；如果会议记录里没有责任人，模型可能猜一个看似合理的人；如果你要求 JSON 输出，模型也可能偶尔漏括号或字段类型错误。

理解这些机制，不是为了否定 LLM 的价值，而是为了知道工程系统该如何使用它。

## 基础解释

这一节不要把 Token、Embedding、Transformer、Attention 当成一组孤立名词来背。更好的学习路线是沿着同一段会议记录走一遍：

> 文本进入模型 -> 被切成 Token -> 转成向量 -> 加入位置信息 -> 在上下文中建立关联 -> 逐步生成输出 -> 由工程系统校验和兜底。

### Token：模型看到的最小文本片段

LLM 不是直接按“字”或“词”理解文本，而是先把文本切分成 Token。Token 可以是一个汉字、一个词的一部分、一个英文单词、标点、空格，具体取决于模型使用的分词器。

比如一句话：

```text
Agent 需要工具。
```

在模型内部，它会被切成一串 Token ID。模型真正处理的是这些 ID，而不是人眼看到的自然语言句子。

Token 对工程系统有三个直接影响。

第一，Token 决定上下文长度。模型一次能处理多少内容，不是按字符数或页数算，而是按 Token 数算。长文档、长对话、工具返回结果都会占用上下文窗口。

第二，Token 影响成本。大多数模型服务都会按输入和输出 Token 计费，或者至少把 Token 作为资源消耗的核心指标。

第三，Token 影响输出稳定性。结构化输出、代码、中文英文混合文本、特殊符号，都可能因为 Token 切分和概率生成带来格式波动。

在 Agent 系统中，Token 不是一个小细节，而是上下文工程、成本控制和性能优化的基础单位。

如果会议记录里有大量寒暄、重复发言和无关讨论，它们同样会占用 Token。工程系统要做的第一步，往往不是“写一个更长的 Prompt”，而是清洗、筛选和压缩上下文。

### Embedding：把文本变成向量

模型不能直接处理自然语言，它需要把 Token 转成向量。可以把向量理解成一组数字，用来表示某个 Token 或某段文本在语义空间中的位置。

Embedding 的直觉是：含义相近的文本，在向量空间中距离也会更近。比如“会议纪要”“会议总结”“会后记录”在语义上接近，它们的向量也可能更接近。

在 LLM 内部，Token Embedding 是模型处理文本的第一步。在 RAG 系统中，文本 Embedding 还会被用来做语义检索：把用户问题和文档片段都转成向量，再找出最相似的文档片段。

这两种 Embedding 都是向量表示，但不要混成一个东西。LLM 内部的 Token Embedding 通常是模型把离散 Token ID 映射为向量表示；RAG 中用于检索的文本 Embedding 通常来自专门的 Embedding 模型，把句子、段落或查询编码成向量。它们粒度不同、用途不同、训练目标也可能不同。

需要注意，Embedding 不是事实本身。它表达的是语义相似性，不保证检索结果一定正确，也不保证模型生成结果一定可信。工程上仍然需要来源引用、重排、过滤和评估。

在会议纪要助手里，Embedding 可以帮助系统找出“和这次会议相关的历史项目资料”。但找到了相似资料，不代表资料一定适用；模型仍然需要来源，系统仍然需要过滤权限和过期信息。

### Transformer：处理上下文关系的核心结构

现代 LLM 多数建立在 Transformer 类架构之上。对本书读者来说，暂时不需要掌握完整公式，但要理解它解决的问题：文本中的每个位置都不是孤立的，它要根据上下文中的其他位置来更新自己的表示。

比如：

```text
苹果发布了新产品。
我今天吃了一个苹果。
```

两个“苹果”字面相同，但上下文不同。第一个更可能指公司，第二个更可能指水果。模型需要根据上下文判断词语含义，这就是上下文建模的重要性。

Transformer 通过多层网络不断更新 Token 表示，让每个 Token 的表示吸收上下文信息。除了 Token Embedding，模型还需要位置信息，让它区分同一个 Token 出现在不同位置时的作用。层数越深，模型理论上越能组合复杂模式，但能力不是只由层数决定，还和参数规模、数据质量、训练方式、架构细节、上下文长度和推理策略有关。

### Attention：让模型关注上下文中的相关信息

Attention 可以理解为一种“相关性计算”机制。模型在处理某个 Token 时，会计算它和上下文中其他 Token 的关系，然后决定哪些信息更重要。

例如，会议记录中有一句：

```text
李四负责在周五前确认上线时间。
```

当模型生成“负责人”时，它需要关注“李四”；当生成“截止时间”时，它需要关注“周五前”；当生成“任务”时，它需要关注“确认上线时间”。

Attention 不是人类意识里的注意力，它是一个数学计算机制。但这个直觉很有用：Attention 帮助模型在上下文中建立关联；模型也会出错，是因为它可能关注了不该关注的内容，或者关键信息根本没有放进上下文。长上下文能力还取决于位置信息、训练方式、架构设计和推理实现，上下文变长不等于模型能稳定使用所有信息。

在 Agent 工程中，这会直接影响上下文组织。你把什么信息放进去、放在什么位置、是否重复、是否冲突，都会影响模型生成。

如果会议纪要里同时出现“李四负责确认上线时间”和“上线时间暂不确定”，模型可能会在生成行动项时混淆。工程上要通过结构化会议记录、来源片段和人工确认减少这种冲突。

### 预训练：从大量数据中学习语言模式

预训练是模型获得基础能力的重要阶段。简化理解，模型会在大量文本数据上学习语言规律，例如词语关系、句子结构、常识模式、代码模式、问答模式。

不同语言模型有不同预训练目标。当前很多自回归生成式 LLM 的核心目标可以简化理解为：给定前文，预测下一个 Token 的概率分布；Masked Token 预测则常见于编码器类语言模型或其他预训练方案。

这个过程让模型学到很多通用能力，但也带来一个边界：模型学习的是数据中的统计模式，不是直接存储一套可验证的事实数据库。它可能学到事实，也可能学到过时信息、错误模式和偏见。

所以，预训练提供能力基础，但不能保证每次回答都正确。

### 微调、指令微调与对齐：让模型更像助手

预训练模型只是学会了语言模式，不一定天然适合和人对话，也不一定会遵循指令。为了让模型更适合实际使用，通常还会经过后续训练或优化，例如微调、指令微调、偏好优化和安全对齐。

可以这样理解：

- 预训练让模型学会语言和知识模式。
- 指令微调让模型更习惯按照用户指令完成任务。
- 对齐让模型更符合人类偏好、安全策略和产品要求。

这些过程会让模型更像一个可用助手，但不会把它变成完全可靠的业务系统。模型仍然可能误解任务、生成不符合格式的内容、拒答过度或回答不完整。

在工程系统里，对齐不能替代权限系统、数据校验、工具沙箱和审计机制。

## 核心原理

### LLM 的基本数据流

可以用下面这个抽象流程理解一次模型调用：

```text
用户输入
  -> Tokenizer：文本切成 Token ID
  -> Embedding：Token ID 转成向量
  -> Position Information：加入位置信息
  -> Transformer Layers：结合上下文更新表示
  -> Output Head：计算下一个 Token 的概率分布
  -> Decoding：选择或采样下一个 Token
  -> 重复生成，直到结束
```

这个流程说明了几个工程事实。

第一，模型生成是逐 Token 进行的。输出越长，推理时间和成本通常越高。

第二，模型每一步都依赖已有上下文。上下文中缺信息、信息冲突、顺序混乱，都可能影响后续生成。

第三，模型输出来自概率分布。即使输入相同，不同采样参数、不同模型版本、不同系统指令，都可能产生不同输出。

第四，模型不是业务系统。它不天然知道你的数据库权限、库存状态、最新工单、当前用户是谁，除非系统把这些信息以受控方式提供给它。

### 为什么模型会“懂语言”

模型并不是像人一样通过生活经验理解语言。更准确地说，它通过大量训练数据学习到了语言中的模式、结构和关联。

比如，模型见过大量类似表达：

```text
请总结以下内容
以下是三点总结
负责人是……
截止时间是……
```

当它看到会议记录和“总结行动项”的指令时，就能生成符合这种任务模式的输出。它看起来像理解，是因为它在高维表示空间中学到了文本和任务之间的复杂关系。

这并不意味着模型没有价值。对工程系统来说，只要模型能稳定把输入转换成有用输出，它就是强大的语言接口。但我们也要记住：这种“理解”不是可验证的事实系统，也不是强一致性的业务逻辑。

### 为什么模型会幻觉

幻觉不是模型“故意说谎”，而是概率生成机制、训练目标、数据分布、对齐方式、上下文质量和解码策略共同影响的结果。模型通常更擅长生成在上下文中看起来合理的内容，而不是逐句验证每句话是否真实。

常见幻觉来源包括：

- 上下文中没有足够信息，模型仍然尝试补全答案。
- 用户问题带有错误前提，模型顺着前提回答。
- 训练数据中存在错误、过时或冲突信息。
- 模型把相似概念混在一起。
- 输出格式要求强，但事实来源不足。
- 工具或检索结果质量差，模型基于错误材料生成。

在 Agent 系统中，幻觉最危险的地方不是“回答错了”，而是它可能驱动后续动作。例如模型错误理解用户意图后调用工具，或者编造一个不存在的 API 参数。

因此，工程系统不能只靠“提示模型不要幻觉”。更可靠的办法是：提供来源、限制工具、校验参数、要求引用、做事实核查、让高风险操作进入人工确认。

### 上下文窗口不是长期记忆

上下文窗口指模型一次调用可以接收的最大 Token 范围。它决定模型“当前这一轮能看到什么”。但上下文窗口不是长期记忆。

如果某条信息没有进入当前上下文，模型就不能可靠使用它。即使以前在对话里出现过，如果系统没有保留并重新传入，模型也可能忘记。

长上下文也不等于无限可靠。上下文越长，成本越高，延迟越高，关键信息越容易被无关内容稀释。工程上需要做取舍：

- 哪些历史对话要保留？
- 哪些内容要摘要？
- 哪些文档片段要检索进入上下文？
- 哪些系统规则必须始终放入？
- 哪些工具结果只在当前步骤有效？

这就是后续“Context Engineering”和“Memory”章节要展开的问题。

### 概率输出与可控性

LLM 输出不是固定函数结果。很多模型调用会涉及采样参数，例如温度、top-p 或类似机制。不同参数会影响输出的确定性和多样性。

工程上可以先建立一个简单直觉：

- 需要稳定格式、抽取字段、执行工具时，更希望输出确定、保守、可校验。
- 需要创意写作、头脑风暴、生成多个方案时，可以接受更多多样性。

但无论参数怎么设置，模型输出都不应该被当成绝对可靠的业务事实。只要输出要进入程序、数据库或工具调用，就需要结构化约束和结果校验。

## 工程实现

本章讲的是原理，但这些原理会直接影响后端系统设计。一个最小 LLM 调用链路可以抽象成这样：

```text
业务请求
  -> 权限和输入校验
  -> 上下文构造
  -> 模型调用
  -> 输出解析
  -> 结果校验
  -> 业务响应或工具调用
  -> 日志、指标和评估记录
```

把这条链路落到后端分层时，可以这样分工：API 层负责认证、租户、输入校验和幂等入口；编排层负责上下文构造、任务状态和人工确认；模型网关负责模型调用、限流、超时、重试、降级和版本记录；工具适配层负责参数校验、权限校验和副作用控制；业务服务层负责事务和真实数据写入；观测层负责 Trace、指标、日志、评估样本和回放记录。

### 把会议纪要助手落到一次后端请求中

一次真实请求可以这样流动：

1. API 层接收会议记录和用户身份，校验文件大小、格式、租户和访问权限。
2. 编排层创建任务，记录 task_id、trace_id、prompt_version 和期望输出 Schema。
3. 上下文层只选择当前任务需要的会议片段、历史项目资料和输出规则，并按 user_id、tenant_id、role/scope 做检索过滤。
4. 模型网关调用合适模型，设置超时、最大输出长度、采样参数和降级策略。
5. 输出解析层校验 JSON、schema_version、字段类型、业务实体存在性、风险等级和是否需要人工确认。
6. 工具适配层在写入任务系统前再次校验权限、幂等键和副作用风险。
7. 观测层记录模型调用、检索材料、工具调用、人工确认和最终响应，供排障、评估和回放使用。

这个例子说明：LLM 原理最终会变成工程边界。Token 影响上下文和成本，概率输出要求校验，幻觉风险要求来源，模型无状态要求后端持久化任务。

### 输入：不要把所有东西都塞给模型

输入模型的上下文应该经过选择和组织。常见输入包括：

- 用户原始问题。
- 系统规则和角色约束。
- 当前任务状态。
- 必要的业务数据。
- 检索到的文档片段。
- 工具调用结果。
- 输出格式要求。

常见错误是把所有历史记录、所有文档、所有工具结果都塞进上下文。这样不仅成本高，还会引入冲突信息，降低模型关注重点。

更好的做法是按任务目标组织上下文：模型完成当前步骤需要什么，就提供什么；不需要的内容尽量不放。

团队和企业系统还要注意权限传播。检索和上下文构造必须带着 user_id、tenant_id、角色和资源范围做过滤，不能先把无权文档放进上下文，再指望模型“不要使用”。一旦无权内容进入上下文，泄露风险已经发生。

### 处理：把模型调用当成不稳定外部依赖

从后端视角看，模型服务不是普通本地函数，而是一个可能慢、可能失败、可能限流、可能返回不合格内容的外部依赖。

因此，模型调用需要工程保护：

- 超时控制：避免请求无限等待。
- 重试策略：只对可重试错误重试，避免重复副作用。
- 降级策略：必要时切换模型、减少上下文或返回部分结果。
- 幂等控制：长任务和工具调用要避免重复执行。
- 速率限制：保护成本和下游模型服务。
- 版本记录：记录模型版本、Prompt 版本和输出格式版本。
- 熔断保护：供应商持续失败时暂停调用，避免故障放大。
- 队列化执行：长任务进入异步队列，避免占满同步请求线程。

这些能力不会出现在论文图里，但会决定系统能不能上线。

### 输出：自然语言给人看，结构化数据给程序用

模型输出可以分成两类。

第一类是给人看的自然语言，比如总结、解释、建议、报告。这类输出需要关注可读性、引用来源、事实准确性和风险提示。

第二类是给程序用的结构化数据，比如 JSON、工具参数、分类结果、抽取字段。这类输出必须校验格式、类型、必填字段、取值范围和权限边界。

例如模型输出：

```json
{
  "assignee": "李四",
  "task": "确认上线时间",
  "deadline": "周五前"
}
```

后端不能因为它看起来像 JSON 就直接写入任务系统。至少要检查：

- `assignee` 是否是系统中的真实用户？
- 当前用户是否有权限给这个人分配任务？
- `deadline` 是否能解析成明确日期？
- 任务内容是否来自会议记录，还是模型补出来的？
- 输出是否符合当前 `schema_version`？
- 是否包含幂等键，避免重复写入？
- 风险等级是否触发审批规则？
- 是否需要人工确认？

这就是 LLM 原理和工程边界的连接点。

### 状态：模型没有你的业务状态

模型本身不会自动知道当前任务执行到哪一步。Agent 系统需要自己维护状态，例如：

- 会话状态：用户当前对话和偏好。
- 任务状态：目标、步骤、进度和最终输出版本。
- 工具执行状态：已调用工具、参数、结果、副作用和幂等键。
- 审批状态：用户确认、人工接管、风险审批结果。
- 成本状态：Token 消耗、模型调用次数、缓存命中和失败次数。

如果状态只存在模型上下文里，系统就很难恢复、审计和重试。生产系统应该把关键状态持久化到数据库、任务表、日志或 Trace 系统中。

### 异常：按来源处理，不要只重试

模型相关异常至少可以分几类：

- 输入异常：上下文过长、用户输入缺失、格式不合法。
- 权限异常：用户无权访问某些数据或调用某些工具。
- 模型异常：超时、限流、服务不可用、返回空内容。
- 输出异常：JSON 解析失败、字段缺失、类型错误。
- 事实异常：模型回答没有来源，或与检索材料冲突。
- 业务异常：下游系统写入失败、任务状态已变化、重复提交。

不同异常不能都靠“再问一次模型”。输入异常应提示用户补充；权限异常应终止并审计；输出格式异常可以尝试修复或重试；高风险事实异常应要求引用或人工确认；业务异常要依赖事务、幂等和补偿机制。

## 适用场景

### 玩具 Demo

在 Demo 阶段，理解 Token、上下文和概率输出就够了。你可以写一个最小脚本，把用户输入发给模型，再观察模型如何生成回答。

Demo 阶段允许没有持久化、没有多用户权限、没有完整评估。它只是帮助你理解输入和输出的关系，不能证明模型真的可靠。

### 个人效率工具

个人工具可以使用 LLM 做总结、改写、翻译、分类、笔记整理和代码辅助。这个阶段要开始关注上下文长度、历史记录、敏感信息和成本。

例如个人笔记助手可以把最近几篇笔记放进上下文，让模型生成摘要。但如果笔记很多，就需要先筛选或摘要，而不是全部塞进去。

个人工具至少要新增基本历史记录、个人数据授权、失败提示和成本提醒。它可以接受偶尔手动修正，但不能默默泄露敏感信息或无限消耗 Token。

### 团队内部工具

团队工具必须把 LLM 当成共享系统能力，而不是个人玩具。需要考虑：

- 不同用户看到的上下文是否不同。
- Prompt 和模型版本是否可追踪。
- 输出错误如何反馈和回放。
- 是否记录 Token、延迟和成本。
- 是否能解释答案来自哪些材料。

比如团队知识库问答系统不能只回答“看起来对”，还要能告诉用户答案依据哪些文档片段。

团队工具需要租户或空间隔离、用户权限、审计日志、Prompt 版本、答案反馈和日志回放。否则一次错误回答很难定位原因，也很难判断是 Prompt、检索、模型还是权限过滤出了问题。

### 企业级系统

企业系统需要把 LLM 调用纳入完整治理。除了效果，还要考虑供应商风险、数据出境、权限隔离、审计日志、SLA、灰度发布、回滚、模型切换和评估集。

在企业级 Agent 中，LLM 原理会影响很多生产决策：上下文如何最小化，哪些内容要脱敏，哪些输出要强校验，哪些任务必须人工确认，哪些模型调用可以缓存或降级。

企业系统还需要 DLP、审批流、合规保留策略、模型和 Prompt 灰度、回滚条件、离线评估集、线上监控阈值和供应商切换预案。

## 不适用场景

LLM 不适合替代确定性业务逻辑。订单金额计算、库存扣减、权限判断、支付状态流转、账务结算，都应该由可测试、可审计的代码和数据库事务完成。

LLM 不适合在没有来源的情况下回答高风险事实问题。医疗、法律、金融、合规等场景，需要来源、审核和专业边界，不能把流畅回答等同于正确答案。

LLM 不适合直接执行高风险操作。删除文件、执行 SQL、退款、发送外部通知、修改生产配置，都需要权限校验、参数校验、风险判断和人工确认。

LLM 不适合承载无限上下文。长文档和长对话需要检索、摘要、分层记忆和信息压缩，不能简单依赖更大的上下文窗口。

LLM 不适合做唯一评估者。模型可以参与辅助评估，但关键业务效果需要测试集、人工抽检、日志回放和真实用户反馈。

## 常见坑与反模式

第一个坑是把 Token 当成字符数。中文、英文、代码、标点和特殊符号的 Token 切分都可能不同。估算上下文和成本时，应该使用模型或平台提供的 Token 统计能力。

第二个坑是把上下文窗口当成记忆。模型只能可靠使用当前输入中提供的信息。长期记忆需要系统自己存储、检索、筛选和更新。

第三个坑是把生成结果当成事实。模型会生成看起来合理的内容，但不保证真实。涉及事实、引用、权限和业务状态时，必须验证。

第四个坑是只靠 Prompt 解决幻觉。Prompt 可以降低风险，但不能替代 RAG、工具校验、来源引用、结构化约束和人工确认。

第五个坑是让模型直接输出工具调用并立即执行。模型可以生成候选参数，但后端必须校验工具是否允许、参数是否合法、操作是否高风险。

第六个坑是没有记录模型调用上下文。上线后如果用户反馈“回答错了”，你需要知道当时使用了哪个模型、哪个 Prompt、哪些上下文、哪些检索材料、哪些工具结果和哪些解码参数。

## 安全、成本与性能考虑

安全方面，LLM 输入和输出都需要边界。不要把用户无权访问的数据放进上下文，不要让模型输出绕过权限系统，不要把敏感 Prompt、原始业务数据和模型输出无控制地写入日志。

成本方面，输入 Token、输出 Token、重试次数、工具调用和模型选择都会影响总成本。工程上要监控每个任务的 Token 使用、模型调用次数、平均输出长度和失败重试成本。

成本控制不只是监控，还要有动作：设置 token budget、max output tokens、用户或租户级 quota、rate limit、缓存策略、模型路由和降级触发条件。超过预算时，系统可以缩短上下文、改用更便宜模型、转异步任务或要求用户确认。

性能方面，LLM 调用通常比普通函数慢，输出越长通常越慢。长任务可以使用异步执行和流式输出；可复用结果可以缓存；低风险简单任务可以使用更便宜或更快的模型；失败场景要有超时、熔断和降级。并发高时，还要考虑队列、优先级、批处理和缓存失效策略。

评估方面，要把模型原理转成指标：

- 上下文是否包含必要信息？
- 输出是否遵循格式？
- 事实是否能找到来源？
- 工具参数是否合法？
- 幻觉率是否可接受？
- 平均延迟和 P95 延迟是否可接受？
- 单次任务成本是否在预算内？

这些指标会在后续可观测性与评估章节进一步展开。

生产评估还需要闭环：从日志中抽取失败样本，构建离线评估集；比较不同模型、Prompt 和上下文策略的效果；设定上线阈值和回滚条件；结合用户反馈、人工抽检和日志回放持续更新评估集。

## 如何评估效果

本章的评估重点不是让读者推导 Transformer 公式，而是能否用原理解释工程现象。

可以用以下问题自测：

- 能否解释 Token 为什么影响上下文、成本和延迟？
- 能否说明 Embedding 为什么适合语义检索，但不等于事实验证？
- 能否用自己的话解释 Attention 解决什么问题？
- 能否说明模型为什么是概率生成，而不是数据库查询？
- 能否解释幻觉的常见来源？
- 能否说明为什么上下文窗口不是长期记忆？
- 能否为一个 LLM 调用链路设计输入校验、输出校验和日志字段？

也可以做一个小型实验：

1. 用同一个问题，分别提供完整上下文和缺失上下文，比较模型回答。
2. 要求模型输出自然语言和 JSON，观察稳定性差异。
3. 故意给一个带错误前提的问题，观察模型是否会顺着错误前提回答。
4. 记录输入长度、输出长度、延迟和错误类型。

如果你能从这些现象反推 Token、上下文、概率输出和校验机制，就说明本章目标达成了。

## 实践任务

1. 最小任务：选择一段 500 字左右的文本，让模型分别做摘要、关键词抽取和 JSON 字段抽取。交付物是一张对比表，记录任务类型、输入长度、输出形式、是否需要校验、错误样例。验收标准是：能说明哪些任务适合自然语言输出，哪些任务必须结构化校验。

2. 工程化任务：设计一个最小 LLM 调用日志结构，至少包含 trace_id、request_id、session_id、user_id、tenant_id、model、prompt_version、schema_version、input_tokens、output_tokens、latency、status、error_code、retry_count、cache_hit、finish_reason。交付物是字段表和一次失败案例复盘。验收标准是：能用这些字段复盘一次失败调用。

3. 进阶任务：为“会议纪要助手”设计一个上下文构造策略，说明哪些内容进入系统指令，哪些来自用户输入，哪些来自检索，哪些来自工具结果，哪些必须留到人工确认。交付物是一张上下文分层表，包含字段来源、是否进入上下文、权限过滤、风险和校验方式。验收标准是：能解释如何控制 Token、降低幻觉并保护敏感信息。

## 从入门到专业

本章不要求你掌握模型数学推导，而是要求你能把模型机制翻译成工程判断：什么会影响上下文，什么会影响输出稳定性，什么必须由后端系统兜底。

- 入门：知道 LLM 是基于 Token、向量、上下文和概率生成工作的，不把它当成魔法。
- 初级：能解释 Token、Embedding、Attention、上下文窗口和幻觉的基本含义。
- 中级：能把 LLM 调用接入后端系统，处理输入、输出、状态、异常和日志。
- 高级：能设计上下文构造、结构化输出校验、重试降级、成本控制和评估方案。
- 专业：能在 Agent 平台中把模型原理转化为运行时策略，包括模型路由、RAG、记忆、工具调用、风险控制和可观测性。

## 本章小结

LLM 的核心不是“会聊天”，而是在给定上下文下逐 Token 生成概率输出。Token 决定输入输出的基本单位，Embedding 把文本变成向量，Transformer 和 Attention 建模上下文关系，预训练提供通用语言能力，指令微调和对齐让模型更适合做助手。

这些机制解释了 LLM 的强大，也解释了它的边界。模型可以总结、改写、抽取、规划和生成代码，但它不是事实数据库，不是权限系统，也不是事务引擎。

对 Agent 工程师来说，理解 LLM 原理的目的不是写论文，而是做正确的系统设计：给模型合适的上下文，限制它能做的动作，校验它的输出，记录它的行为，并在高风险场景中引入工具、来源和人工确认。

下一章会在这个基础上讨论主流模型与能力边界：不同模型为什么表现不同，如何在效果、成本、延迟、上下文、工具调用和部署方式之间做工程选择。
