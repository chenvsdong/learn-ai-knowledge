# 第 1 章：AI、机器学习、深度学习与大模型的关系

## 本章解决什么问题

很多人开始学习 AI Agent 时，会直接跳到 Prompt、RAG、Function Calling、MCP 或某个框架。这样很容易出现一个问题：知道怎么调用模型，却不知道模型在整个 AI 技术体系里处于什么位置，也不知道什么时候该用大模型，什么时候该用传统算法、规则系统或普通后端逻辑。

本章要解决的是一张基础地图：

- AI、机器学习、深度学习、生成式 AI 和大语言模型分别是什么？
- 它们之间是并列关系，还是包含关系？
- 为什么大语言模型会成为 AI Agent 的核心能力来源？
- 在真实工程系统里，什么时候应该让 LLM 参与，什么时候不该让它参与？
- Java 后端工程师如何把这些概念映射到真实系统设计中？

读完本章，读者应该能够用自己的话解释下面这条链路：

> AI 是目标，机器学习是一类方法，深度学习是机器学习中的一类重要方法，生成式 AI 是按“生成内容”划分的一类能力，大语言模型通常处在深度学习模型和生成式 AI 应用的交叉位置，而 AI Agent 是把模型、上下文、工具、记忆、规划和运行时组合起来完成任务的系统。

本章和后续章节的关系也很直接：第 2 章会深入解释 LLM 的基本原理，第 3 章会讨论模型选择和能力边界。没有本章这张地图，后面容易把所有问题都理解成“换一个更强的模型”。

## 一个直观例子

假设你要做一个“会议纪要助手”，用户上传会议录音或文字记录后，希望系统输出会议摘要、待办事项、责任人和风险提醒。

如果只从功能看，这像是一个简单 AI 应用。但从技术组成看，它可能同时用到多种能力：

- 规则系统：判断文件格式、校验用户权限、控制导出模板。
- 传统机器学习：对历史会议类型做分类，或者预测会议优先级。
- 深度学习模型：语音转文字、文本向量化、语义匹配。
- 大语言模型：理解会议内容，生成摘要，抽取行动项，解释风险。
- Agent 系统：在多轮任务中调用转写工具、检索历史项目资料、生成报告、等待人工确认后写入任务系统。

这里最重要的直觉是：LLM 很强，但它不是整个系统。它更像一个具备语言理解和生成能力的“推理与表达核心”。真实产品还需要权限、数据、状态、工具、日志、评估和人工确认。

如果你把所有问题都交给 LLM，系统可能会变得不可控。如果你完全不用 LLM，又很难处理开放语言任务。专业的 Agent 开发不是迷信某一种技术，而是知道每种技术应该站在哪个位置。

## 基础解释

### AI：让机器表现出智能行为的总目标

AI，即人工智能，是一个很大的目标集合。只要系统表现出某种原本需要人类智能才能完成的能力，都可以放进 AI 的大范围里，例如识别图片、理解语言、推荐商品、规划路径、下棋、翻译文本、总结文档。

AI 不是某一种单独算法。它更像一个领域名称，里面包含很多不同路线：

- 基于规则的系统。
- 搜索与规划算法。
- 机器学习模型。
- 深度学习模型。
- 生成式模型。
- Agent 系统。

早期很多 AI 系统并不依赖大规模神经网络。比如专家系统可以通过人工编写规则来判断问题；路径规划可以通过图搜索算法寻找路线。这些方法在规则清楚、状态空间可控的场景里仍然有价值。

回到会议纪要助手，判断文件格式、限制导出模板、检查用户是否能访问某个会议记录，这些都属于更适合用确定性规则完成的部分。它们可以出现在 AI 产品里，但不需要由大模型来决定。

### 机器学习：让系统从数据中学习规律

机器学习是 AI 的一类重要方法。它的核心思想不是把所有规则都写死，而是让模型从数据中学习输入和输出之间的关系。

以监督学习为例，一个简化的机器学习过程可以这样理解：

1. 收集样本数据。
2. 定义输入特征和目标输出。
3. 训练模型，让模型在数据上学习规律。
4. 用新输入让模型做预测。
5. 用评估指标判断效果是否可接受。

例如，给模型大量历史用户行为数据，让它预测用户是否可能点击某个推荐内容；或者给模型大量邮件样本，让它判断一封邮件是不是垃圾邮件。

机器学习的关键不是“模型会思考”，而是它在统计意义上学到了数据中的模式。这个边界很重要，因为模型的能力通常受训练数据、特征质量、任务定义和评估方式影响。除了监督学习，机器学习还包括无监督学习、自监督学习、强化学习等范式，本章先建立地图，不展开各类训练范式。

在会议纪要助手里，如果团队已经积累了大量带标签的会议数据，可以训练或使用一个分类模型来判断会议类型、紧急程度或是否需要同步给某个项目组。这类任务目标比较明确，未必需要 LLM 直接处理。

### 深度学习：用多层神经网络学习复杂表示

深度学习是机器学习中的一类方法，通常使用多层神经网络从数据中学习表示。相比许多依赖人工特征工程的传统机器学习方法，深度学习更擅长从文本、图像、语音、视频这类非结构化数据中自动学习复杂模式。

例如：

- 图像识别模型可以从像素中学习物体特征。
- 语音识别模型可以从音频信号中学习语音和文字之间的关系。
- 文本模型可以从大量文本中学习词语、句子和上下文之间的关系。

深度学习并不等于大语言模型。大语言模型属于深度学习发展出来的重要方向，但深度学习还包括计算机视觉、语音处理、推荐系统、时间序列预测等很多领域。

在会议纪要助手里，语音转文字、文本向量化、语义相似度匹配，通常都可以看作深度学习能力在系统中的位置。它们不一定负责“写总结”，但会影响后续 LLM 能拿到什么输入。

### 生成式 AI：生成新内容的模型能力

生成式 AI 指的是能够生成文本、图片、音频、视频、代码等内容的 AI 能力。它和只做分类、打分、排序的模型不同，更强调“产生一个新的结果”。

在文本领域，大语言模型可以根据上下文生成回答、摘要、代码、计划、解释或结构化内容。在图像领域，生成模型可以根据提示生成图片或编辑图片。

生成式 AI 的价值在于它能处理开放式任务，但风险也来自这里：开放式生成不等于事实正确，也不等于业务安全。生成结果需要校验、约束和评估。

在会议纪要助手里，生成式 AI 体现在“把原始会议内容生成一份摘要、行动项列表或风险说明”。这类输出很有价值，但它不是数据库里的事实记录，最好保留来源引用、置信边界和人工确认入口。

### 大语言模型：以语言为核心接口的通用模型

大语言模型通常指以文本 Token 建模为核心、在大规模语料上训练出的深度学习模型。部分现代模型还扩展到图像、音频等多模态输入输出。它们具备较强的语言理解、生成、总结、转换和对话能力，也可以在某些任务中表现出一定推理能力，但结果仍然需要验证。典型交互方式是输入一段上下文，输出一段文本或结构化结果。

在 Agent 系统中，LLM 的特殊价值不只是“会聊天”，而是它可以充当一种通用的任务理解层：

- 理解用户自然语言目标。
- 把复杂任务拆成步骤。
- 根据上下文生成工具调用意图或候选参数。
- 阅读工具返回结果。
- 生成面向用户或系统的输出。

这就是为什么 LLM 会成为现代 AI Agent 的核心组件之一。但注意，核心组件不等于唯一组件。模型可以提出工具调用意图，是否允许调用、如何校验参数、是否真正执行，必须由 Agent Runtime 和后端策略控制。Agent 还需要记忆、工具、权限、状态机、任务队列、评估和可观测性。

在会议纪要助手里，LLM 可以负责“读懂这场会到底在讨论什么”，也可以把转写文本整理成行动项。但“是否写入任务系统”“是否通知某个成员”“是否访问历史项目资料”，都应该由运行时和权限系统共同控制。

## 核心原理

### 这几个概念的关系

可以用一个简化关系图建立第一眼地图：

```text
AI
├── 规则、搜索、规划等非机器学习方法
├── 机器学习
│   └── 深度学习
│       └── 大语言模型通常属于这里
└── 生成式 AI：按“生成内容”划分的能力类别
    └── 大语言模型通常也是生成式 AI 的重要代表
```

这只是一个帮助理解的抽象模型，不是严格的学科分类图。更准确地说，AI、机器学习、深度学习更像“方法和技术路线”的层级；生成式 AI 更像“能力和任务形态”的分类；LLM 则通常位于深度学习模型和生成式 AI 应用的交叉位置。现实中，生成式 AI 不只包含语言模型，也包括图像、音频、视频等生成模型；AI 也包含不依赖机器学习的规则系统、搜索算法和规划算法。

所以，不要把这张图当成考试式分类题。本书更关心的是：当你设计一个 Agent 系统时，如何把这些能力组合到正确的位置。

更准确的工程理解是：

- AI 关注“系统能不能表现出智能行为”。
- 机器学习关注“能不能从数据中学习规律”。
- 深度学习关注“能不能用多层神经网络学习复杂表示”。
- 生成式 AI 关注“能不能生成新的内容或结果”。
- LLM 关注“能不能以语言为核心接口完成理解、生成和推理任务”。
- Agent 关注“能不能围绕目标持续组织上下文、调用工具、处理状态并完成任务”。

### 从规则到模型，再到 Agent

理解 AI Agent，需要先理解系统能力是如何逐步扩展的。

第一层是规则系统。规则系统适合边界清晰的问题，例如“如果用户没有权限，就拒绝访问”。规则可解释、可测试、可审计，但很难处理开放语言和复杂语义。

第二层是机器学习模型。模型可以从历史数据中学习规律，适合分类、预测、排序、推荐等任务。但很多传统模型需要明确的特征工程和任务定义。

第三层是深度学习模型。深度学习模型更擅长处理非结构化数据，能够从原始文本、图片、语音中学习复杂表示。但它仍然通常围绕特定任务训练和部署。

第四层是大语言模型。LLM 把自然语言变成通用接口，让系统可以处理更开放的输入和输出。它可以总结、改写、解释、抽取、规划，也可以生成代码或结构化数据。

第五层是 Agent。Agent 不是只调用一次模型，而是让模型在一个受控运行时中围绕目标反复执行：读取上下文、决定下一步、调用工具、观察结果、更新状态、输出结论。

### 为什么 LLM 适合做 Agent 的“大脑”

Agent 需要处理的任务往往不是单一预测，而是开放任务。例如：

> “帮我分析这个项目的错误日志，找出可能原因，并生成一个修复建议。”

这个任务包含多个隐含步骤：

- 理解用户目标。
- 判断需要哪些日志、代码或配置。
- 选择工具读取文件或查询系统。
- 把结果组织成上下文。
- 推理可能原因。
- 生成可执行建议。
- 在不确定时请求用户确认。

传统模型通常需要把这些步骤拆成多个专门任务分别训练。LLM 的优势在于，它可以用语言把这些中间步骤连接起来。它既能理解任务描述，也能阅读工具结果，还能生成下一步计划。

但这也是风险来源。LLM 输出的是概率生成结果，不是数据库事实，也不是后端事务。Agent 系统必须给它加上边界：工具权限、参数校验、执行确认、结果验证、日志追踪和失败回退。

### 从概念到判断：遇到需求先问什么

学习这些概念不是为了背名词，而是为了做工程判断。遇到一个 AI 功能需求时，可以先问五个问题：

| 需求特征 | 优先考虑 | 原因 | 主要风险 |
| --- | --- | --- | --- |
| 规则明确、结果必须确定 | 普通代码或规则系统 | 可测试、可审计、成本低 | 不适合开放语言理解 |
| 有历史样本、目标清晰 | 传统机器学习或专用模型 | 适合分类、预测、排序 | 依赖数据质量和评估集 |
| 输入是语音、图像、长文本等非结构化数据 | 深度学习模型或专用服务 | 擅长学习复杂表示 | 需要处理模型误差和数据漂移 |
| 需要理解、总结、改写、抽取开放语言 | LLM | 自然语言接口灵活 | 可能幻觉、格式不稳、成本较高 |
| 需要多步执行、调用工具、维护状态 | Agent Runtime + LLM + 工具 | 能把理解、行动和反馈串起来 | 权限、状态、异常和审计复杂 |

会议纪要助手正好覆盖这些问题：文件校验适合规则，会议类型判断可以用分类模型，语音转写和向量检索依赖深度学习，摘要和行动项适合 LLM，多步调用工具和等待确认则进入 Agent 范畴。

## 工程实现

本章不要求实现一个完整 Agent，但需要建立工程映射。下面是一个概念性的 Agent 后端结构，用来说明 AI、ML、DL 和 LLM 在系统中的位置。

```text
用户请求
  -> API 层：认证、限流、参数校验
  -> 任务层：创建任务、记录状态、选择处理流程
  -> 上下文层：组装用户输入、历史记录、业务数据
  -> 模型层：调用 LLM、Embedding 模型或分类模型
  -> 工具层：搜索、数据库、文件、业务系统 API
  -> 校验层：结构化校验、权限校验、事实校验
  -> 输出层：返回答案、写入任务系统、记录日志
```

在这个结构里，不同 AI 技术的位置不同。

规则系统通常放在 API 层、权限层、校验层和工作流控制中。比如用户是否能读取某个项目文档，不能交给 LLM 自己判断，而应该由后端权限系统决定。

传统机器学习模型常放在明确任务节点中。例如风险评分、意图分类、排序、推荐、异常检测。这些任务输入输出稳定，适合用专门模型或业务算法处理。

深度学习模型常用于非结构化数据处理。例如语音转文字、图片理解、文本向量化、语义检索。它们经常作为 Agent 的感知能力或知识检索能力。

LLM 常放在任务理解、内容生成、复杂转换、计划生成和工具调用意图生成中。它擅长把自然语言任务转成步骤，也擅长把多源信息整合成可读输出。

Agent Runtime 则负责把这些能力串起来。它不应该只是一个“调用 LLM 的函数”，而应该管理状态、步骤、工具调用、异常、重试、人工确认和审计记录。

在生产系统里，这条链路通常还会变成一个请求生命周期：同步 API 接收请求并创建任务，任务进入队列或工作流执行器，运行时持久化每一步状态，模型和工具调用按策略重试或降级，用户可以取消任务或补充确认，超时任务要能回收，重复提交要能通过幂等键识别。也就是说，Agent 后端不是“一次 HTTP 请求里把所有事做完”，而是一套可恢复、可追踪、可治理的任务系统。

入口鉴权也不够。Agent 每次读取数据、检索文档、调用工具、写入业务系统时，都要基于用户、租户、资源和操作做二次授权。尤其在团队和企业系统中，文档级权限、工具级权限、操作级权限、临时授权和人工确认要一起设计。

### 一个后端视角的最小流程

下面是伪代码，用来表达职责划分，不对应任何具体框架 API：

```text
receive(userRequest)
  authenticate(user)
  validateInput(userRequest)

  task = createTask(userRequest)
  context = buildContext(user, task, businessData)

  intent = classifyIntent(context)

  if intent.requiresStructuredData:
    result = callBusinessService(context)
  else if intent.requiresOpenReasoning:
    result = callLLM(context)
  else:
    result = applyRules(context)

  checkedResult = validateResult(result)
  recordTrace(task, context, result, checkedResult)
  return checkedResult
```

这段伪代码表达一个关键原则：LLM 是能力之一，不是所有分支的唯一入口。专业的 Agent 系统会根据任务类型选择最合适的处理方式。

### 输入、处理、输出、状态与异常

真实后端系统至少要把五件事说清楚。

输入是什么：用户自然语言、上传文件、业务对象 ID、历史会话、权限上下文、系统规则。

处理过程是什么：哪些步骤由规则执行，哪些步骤由模型执行，哪些步骤由工具执行，哪些步骤需要人工确认。

输出是什么：自然语言答案、JSON 结构、数据库写入、工单、报告、任务状态变更。

状态在哪里：会话状态、任务状态、工具调用状态、模型请求状态、用户确认状态，需要持久化到可追踪的位置。

异常怎么处理：用户输入错误通常应该直接返回可修正提示；权限不足应该终止并记录审计；模型超时、供应商限流可以重试、切换模型或降级；工具调用失败要区分是否已经产生副作用；结果校验失败可以修复、重试或转人工；业务写入失败必须依赖事务、幂等和补偿机制。

这就是从“会调用模型”到“能做 AI Agent 后端”的分水岭。

## 适用场景

### 玩具 Demo

在 Demo 阶段，可以直接让用户输入问题，然后把问题发给 LLM，观察模型回答。这个阶段的目标是理解自然语言输入和模型输出之间的关系。

但 Demo 不能证明系统可生产。它通常没有权限、状态、日志、成本控制和评估机制。

### 个人效率工具

个人工具可以开始组合多种能力。例如用 LLM 总结个人笔记，用 Embedding 做语义检索，用规则整理文件路径。

这个阶段要开始关注长期使用体验：历史记录是否可找回，输出是否稳定，敏感信息是否会被错误发送到外部模型，成本是否可控。

### 团队内部工具

团队工具必须考虑多人协作。比如一个团队知识库问答 Agent，需要处理不同成员的文档权限、引用来源、回答质量、错误反馈和日志追踪。

在这个阶段，不能只问“模型回答得好不好”，还要问“谁问了什么、用了哪些数据、调用了哪些工具、结果是否可审计”。

### 企业级系统

企业系统要把 Agent 当成生产系统的一部分。除了效果，还要考虑安全、合规、成本、SLA、回滚、灰度发布、模型供应商切换和数据治理。

在企业级系统里，LLM 往往不是直接面对所有业务数据，而是通过受控接口、权限系统、审计链路和安全策略间接访问。

常见上线形态也会更复杂：Agent API 负责接入业务入口，异步 Worker 或任务队列承载长任务，模型网关统一管理模型供应商、限流、成本和降级，工具服务封装数据库、文件和企业 API，配置中心控制 Prompt 版本、工具开关和灰度策略，评估系统负责回放测试和上线前验收。

可以用下面的最低要求来判断系统成熟度：

| 层级 | 最低工程要求 | 不应承担的风险 |
| --- | --- | --- |
| 玩具 Demo | 能跑通模型调用和基本流程 | 不承诺稳定性、权限和长期可用 |
| 个人效率工具 | 有历史记录、基本成本控制和失败提示 | 不处理多人权限和审计责任 |
| 团队内部工具 | 有用户身份、数据隔离、日志、反馈和权限控制 | 不直接执行高风险生产操作 |
| 企业级系统 | 有审计、SLA、灰度、回滚、评估、治理和安全策略 | 不让模型绕过事务、权限和合规边界 |

## 不适用场景

不是所有问题都适合用 LLM 或 Agent。

如果任务规则明确、输出确定、成本敏感，普通代码或规则系统可能更好。例如订单金额计算、权限判断、库存扣减、支付状态流转，这些任务不应该交给概率模型决定。

如果任务有强一致性要求，必须依赖数据库事务、幂等控制和可验证逻辑。LLM 可以辅助解释或生成建议，但不能替代核心事务逻辑。

如果任务只需要简单分类或排序，传统机器学习或业务规则可能更便宜、更稳定、更容易评估。

如果数据高度敏感，而系统还没有完成权限隔离、脱敏、审计和供应商风险评估，就不应该急着把数据接入外部大模型。

如果用户期望的是可证明正确的答案，而系统又无法提供可靠来源、验证机制或人工确认，那么直接使用生成式模型会带来较高风险。

## 常见坑与反模式

第一个坑是把 AI 等同于 LLM。截至 2026 年，LLM 是 AI Agent 工程中非常重要的一类模型，但 AI 系统还包括规则、搜索、规划、传统机器学习、深度学习和工程系统。把 AI 简化成 LLM，会导致架构设计过度依赖模型。

第二个坑是把 Agent 等同于聊天机器人。聊天机器人主要关注对话体验，而 Agent 还要关注目标、状态、工具、记忆、执行和验证。

第三个坑是让 LLM 判断权限。权限必须由确定性的后端系统控制。LLM 可以解释“为什么没有权限”，但不能成为权限判定源。

第四个坑是用生成结果直接驱动高风险操作。比如模型生成一个数据库操作、退款指令或删除命令后直接执行。这类操作至少需要参数校验、权限校验、风险判断和人工确认。

第五个坑是没有评估就上线。生成式 AI 的输出看起来流畅，不代表准确、稳定或安全。越是自然的回答，越容易让人忽略验证。

第六个坑是只优化 Prompt，不设计系统。Prompt 很重要，但 Agent 的稳定性还依赖上下文工程、工具设计、状态管理、错误处理和观测体系。

## 安全、成本与性能考虑

安全方面，最重要的是边界。模型输入不能随意包含敏感数据，模型输出不能绕过权限系统，工具调用不能越权执行。对于团队和企业系统，需要记录用户身份、数据来源、工具调用和关键决策。Prompt、上下文和模型输出不应无约束全量落日志，而应该根据敏感级别做脱敏、采样、分级存储和访问控制。

成本方面，LLM 调用通常比普通后端逻辑更贵，也更容易受上下文长度影响。工程上应该区分任务类型：简单任务用规则或小模型，复杂语言推理任务再调用更强的模型。这里是工程经验判断，不是绝对规则。

性能方面，模型调用通常比本地函数和数据库查询慢。用户体验需要考虑异步任务、流式输出、缓存、超时、降级和重试。对于长任务，应该把执行状态持久化，而不是依赖一次 HTTP 请求完成所有工作。

评估方面，不能只看“回答像不像”。应该记录任务成功率、用户反馈、人工接管率、工具失败率、输出格式错误率、平均延迟、成本趋势和高风险操作拦截情况。

最小可观测字段可以包括：trace_id、task_id、user_id、tenant_id、session_id、model、prompt_version、tool_name、latency、token_usage、cost_estimate、risk_level、decision、error_code 和 fallback_action。字段不是越多越好，关键是能复盘“谁在什么权限下，用了什么数据和工具，为什么得到这个结果，失败时系统做了什么”。

这些指标不需要在第一天全部做完，但从系统设计开始就要给它们留位置。

## 如何评估效果

本章是基础概念章节，评估重点不是模型准确率，而是读者和系统设计者是否建立了正确分类。

可以用以下问题自测：

- 能否说明 AI、机器学习、深度学习、生成式 AI、LLM 和 Agent 的关系？
- 能否举例说明一个系统里哪些部分应该用规则，哪些部分可以用模型？
- 能否说明为什么 LLM 适合做 Agent 的任务理解和生成核心？
- 能否说明为什么权限、事务和高风险操作不能交给 LLM 自行决定？
- 能否画出一个最小 Agent 后端的数据流？
- 能否为一个 AI 功能设计基本日志和评估指标？

对于实际项目，可以用一个小型设计评审来验证：

1. 选一个具体功能，例如知识库问答、日志分析或会议纪要。
2. 标出输入、处理、输出、状态、异常。
3. 标出规则、传统模型、深度学习模型、LLM 和工具分别负责什么。
4. 标出哪些环节需要权限、审计、评估和人工确认。

如果这些问题回答不清楚，说明还没有真正理解 AI Agent 的工程边界。

## 实践任务

1. 最小任务：画一张概念关系图，解释 AI、机器学习、深度学习、生成式 AI、LLM 和 Agent 的关系。要求能用非技术语言讲给一个产品经理听。验收标准是：能说明 AI/ML/DL 的层级关系，也能说明生成式 AI 是能力分类，LLM 不是整个 AI。

2. 工程化任务：选择一个你熟悉的业务功能，例如客服问答、日志分析、合同摘要或报表解读，拆分其中的规则逻辑、模型逻辑、工具调用和人工确认环节。验收标准是：至少标出输入、输出、状态、权限、异常和日志字段。

3. 进阶任务：为这个功能设计一个最小 Agent 后端流程，写清楚输入、状态、工具、输出、异常处理、日志字段和评估指标。可以使用伪代码或流程图，不需要绑定具体框架。验收标准是：能说明哪些步骤可以重试，哪些步骤必须终止，哪些步骤需要人工确认。

## 从入门到专业

- 入门：知道 AI、机器学习、深度学习、生成式 AI、LLM 和 Agent 不是同一个概念，能说出主要关系和边界。
- 初级：能用 LLM 做一个最小问答或总结 Demo，理解输入上下文和输出之间的关系。
- 中级：能把 LLM 接入真实后端流程，区分规则、模型、工具和业务系统的职责。
- 高级：能处理权限、状态、异常、日志、评估、成本和性能问题，不把模型回答直接当成业务事实。
- 专业：能在完整 Agent 平台中设计模型层、工具层、记忆层、运行时、治理和评估体系，并能判断什么时候不该使用 Agent。

## 本章小结

本章最重要的结论是：AI Agent 不是凭空出现的新魔法，而是建立在 AI、机器学习、深度学习和大语言模型等能力之上的工程系统。

AI 是目标，机器学习是方法，深度学习是重要技术路线，生成式 AI 提供内容生成能力，LLM 让自然语言成为通用接口，而 Agent 把模型能力放进一个有目标、有状态、有工具、有边界的执行系统里。

对于后端工程师来说，学习 AI Agent 不能只学“怎么调模型”。更关键的是学会判断：哪些事情适合交给模型，哪些事情必须留在确定性系统里，模型输出如何验证，工具调用如何受控，任务状态如何追踪。

有了这张地图，下一章就可以进一步进入 LLM 的内部：Token、Embedding、Transformer、Attention、预训练、对齐和幻觉，理解大语言模型为什么能工作，也理解它为什么会失败。
