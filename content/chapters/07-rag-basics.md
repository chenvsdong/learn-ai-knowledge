# 第 7 章：RAG 基础

## 本章解决什么问题

第 4 章讲 Prompt，第 5 章讲 Context，第 6 章讲结构化输出。到这里，我们已经知道如何给模型清晰任务、如何组织上下文、如何让输出进入程序。但还有一个核心问题没有解决：模型不知道你的私有知识、最新业务规则、内部文档、项目代码和实时数据时，怎么办？

RAG，也就是 Retrieval-Augmented Generation，解决的是“先从外部知识源检索相关材料，再让模型基于这些材料生成回答”的问题。它不是让模型永久学会新知识，也不是把整个知识库塞进 Prompt，而是在每次请求时按需取材料。

本章要回答：

- 为什么需要 RAG，而不是直接依赖模型参数知识？
- RAG 和长上下文、微调、记忆系统有什么区别？
- 文档加载、清洗、切分、索引、检索、重排、生成分别做什么？
- Embedding、向量数据库、关键词检索、混合检索分别解决什么问题？
- Java 后端工程师如何实现一个最小 RAG 系统？
- 权限、引用、版本、缓存、日志和评估应该放在哪里？
- 什么时候不该用 RAG？

截至 2026-05，RAG 的具体实现会因模型供应商、Embedding 模型、向量数据库、搜索引擎、框架和企业数据系统不同而变化。本章不写死某个 SDK 方法名、向量维度、top-k 数字或性能指标，而是建立可迁移的工程方法。具体 API、参数和限制必须以当前官方文档和本地项目为准。

读完本章，读者应该能画出一个最小 RAG 后端架构，知道一次问答中数据如何从文档进入索引、从索引进入上下文、从上下文进入模型答案，并能判断这个答案是否有来源、是否越权、是否可复盘。

## 一个直观例子

主线交接：第 6 章已经保证模型输出能绑定证据并被程序校验；本章新增能力是把团队知识库文档安全地入库、检索并注入上下文。下一章会在这个基础上处理查询改写、多路召回和重排。

继续使用知识库问答助手。团队内部有一份上线检查文档：

```text
文档：上线检查清单
版本：2026-05-20

上线前必须完成：
1. 权限过滤测试。
2. 安全评审。
3. 至少 20 条常见问题评估样本。
4. 至少 5 条越权访问评估样本。
```

用户问：

```text
知识库问答助手上线前还缺什么？
```

如果只把问题发给模型，模型可能根据通用经验回答：

```text
上线前需要测试、监控、灰度发布和用户反馈。
```

这不一定错，但它没有回答团队内部文档里的具体要求。一个 RAG 系统会先做几件事：

1. 根据用户问题构造检索查询。
2. 在用户有权限访问的知识库范围内检索相关文档。
3. 找到“上线检查清单”相关片段。
4. 把通过权限校验的片段、来源和版本放进上下文。
5. 要求模型只基于这些材料回答，并给出引用。

被过滤的材料、过滤原因和权限判断结果应该进入受控安全审计日志，不应该进入模型输入或普通 trace。否则模型虽然没有看到原文，用户也可能从被拒绝的文档 ID、标题、路径或权限规则里推断出不该知道的信息。

更可靠的回答可以是：

```text
根据《上线检查清单》（版本 2026-05-20），上线前需要确认四项：
1. 权限过滤测试。
2. 安全评审。
3. 至少 20 条常见问题评估样本。
4. 至少 5 条越权访问评估样本。

如果这些项目尚未完成，不能把上线时间写成确定结论。
```

RAG 的价值不是让模型“更会聊天”，而是让模型的回答落在可追溯、可授权、可更新的外部材料上。

## 基础解释

### RAG 是什么

RAG 可以拆成三个动作：

```text
Retrieval：检索相关材料
Augmented：把材料增强到当前上下文
Generation：让模型基于材料生成回答
```

在工程系统里，它通常表现为一条链：

```text
用户问题
  -> 查询理解
  -> 检索知识库
  -> 选择相关片段
  -> 组装上下文
  -> 调用模型
  -> 生成带来源的回答
```

RAG 的关键不是“用了向量数据库”，而是“模型回答时能看到与当前问题相关、且允许被看到的外部材料”。

### RAG 不等于向量数据库

很多初学者会把 RAG 等同于“Embedding + 向量数据库”。这是常见误解。

Embedding 和向量检索是 RAG 的常见实现方式，但不是唯一方式。一个 RAG 系统可以使用：

- 关键词检索，例如倒排索引、BM25。
- 向量检索，例如基于 Embedding 的语义相似度。
- 混合检索，例如关键词 + 向量 + 元数据过滤。
- 数据库查询，例如按产品、租户、日期、标签过滤业务记录。
- 图检索，例如从知识图谱中查实体关系。

向量数据库解决的是“如何快速找语义相近的片段”。RAG 解决的是“如何把外部材料安全、准确、可评估地带入模型生成过程”。两者有关，但不能画等号。

### RAG、微调、长上下文和记忆的区别

这几个概念经常被混在一起：

| 能力 | 解决什么 | 适合什么 | 不适合什么 |
| --- | --- | --- | --- |
| RAG | 按需检索外部知识 | 内部文档、最新规则、知识库问答 | 让模型永久学会某种风格或能力 |
| 微调 | 改变模型行为或任务模式 | 固定格式、领域表达、分类风格 | 高频更新的事实知识 |
| 长上下文 | 单次输入更多材料 | 长文档阅读、一次性分析 | 替代检索、权限和筛选 |
| 记忆系统 | 保存用户或任务的长期状态 | 用户偏好、历史决策、项目状态 | 无审查地保存所有聊天内容 |

如果知识经常变化，RAG 通常比微调更适合；如果是输出风格、分类口径或特定任务模式，微调可能更合适；如果只是一次性阅读一份长材料，长上下文就可能够用；如果要跨会话保存用户偏好或任务状态，需要第 9 章的记忆系统。

### RAG 的最小组成

一个最小 RAG 系统至少有两条链路：

```text
离线链路：文档 -> 清洗 -> 切分 -> 索引
在线链路：问题 -> 检索 -> 组装上下文 -> 生成 -> 校验
```

离线链路决定“知识如何进入系统”。在线链路决定“知识如何进入一次模型请求”。

这两个链路都很重要。只关注在线问答，不处理文档质量，检索会召回脏材料；只关注索引构建，不处理权限和引用，回答可能越权或不可验证。

## 核心原理

### 原理一：模型参数知识不等于业务知识

模型训练时学到的是参数化知识。它可能知道通用编程概念、公开技术资料和常见业务模式，但不知道你公司昨天更新的上线规则、某个项目的权限配置、内部接口文档或最新工单状态。

RAG 把知识从“模型内部参数”移到“外部可管理数据源”。这样有几个好处：

- 知识可以更新，不必重新训练模型。
- 来源可以追踪，回答能给引用。
- 权限可以控制，不同用户看到不同材料。
- 版本可以管理，过期文档可以下线。
- 错误可以修复，改文档或索引即可回归测试。

这也是 RAG 在企业场景中很重要的原因：企业知识不是静态百科，而是有权限、有版本、有生命周期的数据资产。

### 原理二：切分决定检索上限

文档进入 RAG 系统前通常要被切成片段。片段太大，检索结果可能包含大量无关内容；片段太小，关键上下文可能被拆散。

例如一段上线检查文档：

```text
上线前必须完成权限过滤测试、安全评审、20 条常见问题评估样本和 5 条越权访问评估样本。若安全评审未通过，只允许内部测试组使用。
```

如果切得太小：

```text
片段 1：上线前必须完成权限过滤测试。
片段 2：安全评审。
片段 3：20 条常见问题评估样本。
片段 4：若安全评审未通过，只允许内部测试组使用。
```

模型可能看不到“这些要求共同构成上线门槛”。如果切得太大，把整份几十页文档作为一个片段，检索结果又会过长、成本高、噪声多。

切分不是纯文本处理问题，而是语义边界问题。标题、段落、表格、代码块、列表、版本说明、权限标签都可能影响切分策略。

### 原理三：检索相关不等于回答正确

检索系统找到了相关片段，并不代表模型答案一定正确。还要处理：

- 片段是否是最新版本。
- 片段是否来自可信来源。
- 用户是否有权限查看。
- 片段是否完整表达了条件和例外。
- 多个片段之间是否互相冲突。
- 模型是否真的引用了入选片段。

例如检索到一份旧文档写着“上线前只需 10 条评估样本”，而新文档写着“至少 20 条常见问题和 5 条越权问题”。如果系统没有版本控制，模型可能引用旧规则。

RAG 的可靠性来自检索、排序、上下文组装、引用校验和评估的共同作用，不是单个向量相似度分数。

### 原理四：RAG 是上下文工程的一部分，但有独立链路

第 5 章讲过，Context Engineering 负责决定模型能看到什么。RAG 提供了一类重要上下文来源：外部知识库。

但 RAG 不是简单的“把检索结果塞进上下文”。它有自己的工程链路：

```text
数据源管理
  -> 文档解析
  -> 清洗和切分
  -> 元数据标注
  -> 索引构建
  -> 查询理解
  -> 召回
  -> 重排
  -> 上下文注入
  -> 引用和评估
```

所以可以这样理解：Context Engineering 关心“本次请求要看哪些材料”；RAG 关心“外部知识如何被检索、排序、注入和评估”。

### 原理五：权限必须进入检索链路

RAG 的安全边界不能只靠 Prompt。

错误做法是：

```text
先检索全库，把结果放进上下文，再告诉模型：不要泄露无权限内容。
```

正确做法是：

```text
先确定用户、租户、项目和资源权限
  -> 只在允许范围内检索
  -> 召回后再次校验 ACL
  -> 被过滤材料只进入审计日志，不进入模型输入
```

模型不应该看到用户无权访问的内容。即使模型“承诺不说”，内容已经进入上下文时，安全边界就已经破了。

## 工程实现

### 最小 RAG 架构

一个 Java 后端里的最小 RAG 架构可以这样拆：

```text
文档管理端
  -> Document Loader
  -> Document Parser
  -> Chunker
  -> Metadata Enricher
  -> Index Writer

问答服务端
  -> Auth / Permission Service
  -> Query Planner
  -> Retriever
  -> Reranker
  -> Context Assembler
  -> Model Gateway
  -> Citation Validator
  -> Trace Logger
```

对应职责：

| 模块 | 责任 | 常见输出 |
| --- | --- | --- |
| Document Loader | 从文件、网页、数据库、对象存储加载原始资料 | raw_document |
| Document Parser | 解析 PDF、Markdown、HTML、Word、代码等格式 | parsed_document |
| Chunker | 按语义边界切分片段 | chunks |
| Metadata Enricher | 添加来源、版本、租户、权限、时间、标签 | chunk_metadata |
| Index Writer | 写入搜索引擎、向量库或数据库索引 | index_record |
| Query Planner | 理解用户问题，生成检索查询和过滤条件 | retrieval_query |
| Retriever | 从索引召回候选材料 | candidate_chunks |
| Reranker | 对候选材料重新排序和筛选 | selected_chunks |
| Context Assembler | 组装可注入模型的上下文 | rag_context |
| Citation Validator | 检查答案引用是否来自入选材料 | citation_result |
| Trace Logger | 记录检索、过滤、版本、token 和引用 | rag_trace |

刚开始可以把这些模块写在一个服务里，但职责要分清。否则后期排查“为什么答案错了”时，会分不清是文档解析错、切分错、检索错、权限错、重排错，还是模型生成错。

### 文档入库链路

文档入库不是把文件原文直接丢进向量库。生产系统至少要做：

1. 读取原始文档。
2. 解析正文、标题、表格、代码块和链接。
3. 清理页眉页脚、目录、重复导航、乱码和无关模板。
4. 保留来源、版本、更新时间、作者、所属项目、租户和权限标签。
5. 按语义边界切分。
6. 为每个片段生成可检索表示。
7. 写入索引，并记录索引版本。
8. 校验源文档版本和索引记录是否一致。
9. 在文档更新、删除或权限变化时触发重建、失效或回滚。

一个抽象的 chunk 可以长这样：

```json
{
  "chunk_id": "kb-checklist-20260520-003",
  "document_id": "kb-checklist",
  "title": "上线检查清单",
  "section": "上线门槛",
  "content": "上线前必须完成权限过滤测试、安全评审、20 条常见问题评估样本和 5 条越权访问评估样本。",
  "source_uri": "knowledge://project-a/release-checklist",
  "version": "2026-05-20",
  "tenant_id": "tenant-a",
  "project_id": "project-a",
  "visibility": "team-a",
  "updated_at": "2026-05-20T10:00:00+08:00"
}
```

注意：`visibility`、`tenant_id`、`project_id` 不是装饰字段，而是检索时必须使用的权限和过滤条件。

入库链路还要把上传文件当成不可信输入处理。支持 PDF、Word、HTML、网页、代码包或图片 OCR 时，生产系统至少要补一层入库安全和作业状态：

- 上传校验：检查文件大小、页数、MIME 类型、扩展名、压缩包层级和租户归属，避免压缩炸弹或越权挂载。
- 恶意内容扫描：对宏、脚本、可执行附件、异常链接和已知恶意文件做扫描；不确定时进入隔离区。
- 解析沙箱：文档解析器、OCR 和 HTML 清洗应在受限环境中运行，限制 CPU、内存、网络访问和文件系统权限。
- 外链控制：默认不要让解析器自动抓取外部链接；需要抓取时要走 allowlist、超时、大小限制和审计。
- 失败状态：区分 `parse_failed`、`unsupported_type`、`virus_suspected`、`permission_unbound`、`ocr_low_quality`、`dead_letter` 等状态。
- 入库隔离：解析完成、权限绑定、版本校验和安全检查通过前，材料不能进入可召回索引。

这些步骤不属于“让模型更聪明”，但决定了 RAG 系统能不能安全接收真实企业文档。否则，一个恶意文件或解析器漏洞就可能绕过后面的检索和模型治理。

### 索引生命周期

RAG 的生产问题经常不是“第一次怎么入库”，而是“文档变化后索引是否仍然可信”。一个最小索引生命周期至少要覆盖：

| 事件 | 系统动作 | 失败处理 |
| --- | --- | --- |
| 新增文档 | 解析、切分、写入索引，记录 `source_version` 和 `index_version` | 入库任务可重试，重复提交靠幂等键去重 |
| 更新文档 | 标记旧 chunk 失效，生成新 chunk，切换到新索引版本 | 新版本未完成前继续使用旧版本或进入降级 |
| 删除文档 | 删除结构化记录、向量记录、缓存和可召回引用 | 删除失败要告警，禁止继续召回已删除材料 |
| 权限变化 | 重新计算元数据过滤字段，必要时重建相关索引 | 权限不确定时 fail closed，不把材料放进上下文 |
| 批量重建 | 新旧索引并行，评估通过后切流 | 保留回滚到旧索引版本的能力 |

每个入库任务都应该有任务状态，例如 `pending`、`parsing`、`indexed`、`failed`、`deleted`、`stale`。线上问答时要能知道答案基于哪个 `index_version`、哪个 `source_version`。如果源文档已经更新但索引仍是旧版本，系统要么明确标注可能过期，要么拒绝给出确定结论。

### 切分策略

基础切分可以从简单规则开始：

- Markdown 按标题和段落切。
- FAQ 按问题和答案切。
- API 文档按接口或字段组切。
- 代码按类、函数、配置块切。
- 表格按行组或业务对象切。

切分时要保留上下文信息。例如某个段落属于哪个标题、哪个文档、哪个版本。否则检索到片段后，模型可能看不到片段的语义位置。

一个实用做法是“正文片段 + 元数据标题”一起进入检索：

```text
title: 上线检查清单
section: 上线门槛
content: 上线前必须完成权限过滤测试、安全评审、20 条常见问题评估样本和 5 条越权访问评估样本。
```

第 8 章会更深入讲 chunk 大小、重叠、层级切分、多路召回和重排。本章先记住：切分影响召回，召回影响上下文，上下文影响答案。

### 检索方式

常见检索方式有三类：

| 方式 | 优点 | 风险 |
| --- | --- | --- |
| 关键词检索 | 对精确术语、错误码、接口名、字段名友好 | 不理解语义改写 |
| 向量检索 | 能找到语义相近内容 | 可能召回语义相近但事实不匹配的片段 |
| 混合检索 | 兼顾精确匹配和语义匹配 | 实现复杂，需要评估权重和排序 |

例如用户问“上线前缺哪些评估样本”，关键词检索能命中“评估样本”；向量检索可能命中“验收用例”“测试集”；混合检索可以把两类结果合并，再用排序或重排选择更可靠的片段。

### 在线问答链路

一次在线 RAG 请求可以这样走：

```text
UserQuestion
  -> 生成 trace_id
  -> 识别用户、租户、项目和权限
  -> 判断是否需要检索
  -> 构造 retrieval_query 和 metadata_filter
  -> 检索候选片段
  -> ACL 二次校验
  -> 去重、排序、重排
  -> 裁剪成 rag_context
  -> 调用模型生成答案
  -> 校验引用和输出结构
  -> 记录 trace
```

一个在线请求对象可以抽象成：

```json
{
  "trace_id": "rag_20260528_001",
  "tenant_id": "tenant-a",
  "user_id": "u_123",
  "project_scope": ["project-a"],
  "question": "知识库问答助手上线前还缺什么？",
  "retrieval": {
    "query": "知识库问答助手 上线前 评估样本 权限过滤 安全评审",
    "filters": {
      "tenant_id": "tenant-a",
      "project_id": "project-a",
      "visibility": ["public", "team-a"]
    },
    "top_k": "configured-by-system"
  }
}
```

`top_k` 这里故意不写固定数字。它不是越大越好，要根据文档规模、片段质量、模型上下文窗口、成本和评估结果决定。

### 上下文注入格式

检索片段进入模型前，应该带上来源和边界：

```text
[retrieved_context]
chunk_id: kb-checklist-20260520-003
title: 上线检查清单
version: 2026-05-20
source: knowledge://project-a/release-checklist
content:
上线前必须完成权限过滤测试、安全评审、20 条常见问题评估样本和 5 条越权访问评估样本。
[/retrieved_context]
```

同时要明确告诉模型：

```text
请只基于 retrieved_context 中的材料回答。
如果材料不足，请说明缺少什么信息。
回答中必须引用 chunk_id。
不要把未检索到的通用经验当成本文档结论。
```

这些指令不能替代后端校验。后端仍要检查答案引用的 `chunk_id` 是否来自本次入选片段，是否属于当前用户可见范围。

### Java 后端伪代码

下面是一个不绑定 Spring AI、LangChain4j 或特定向量库的伪代码：

```java
class RagService {
    RagAnswer answer(RagQuestion question) {
        Trace trace = traceService.start(question);

        PermissionScope scope = permissionService.resolveScope(
            question.userId(), question.tenantId(), question.projectId()
        );

        RetrievalQuery query = queryPlanner.plan(question.text(), scope);

        List<Chunk> candidates = retriever.search(query.withMetadataFilter(scope));

        List<Chunk> allowed = candidates.stream()
            .filter(chunk -> permissionService.canRead(scope, chunk))
            .toList();

        List<Chunk> selected = reranker.select(question.text(), allowed);

        RagContext context = contextAssembler.render(selected);

        ModelOutput output = modelGateway.generate(question.text(), context);

        CitationResult citation = citationValidator.validate(output, selected);

        traceService.finish(trace, query, traceSafe(candidates), selected, citation);

        return answerMapper.toRagAnswer(output, citation);
    }
}
```

重点不是类名，而是顺序：先确定权限，再把权限范围变成检索过滤条件，召回后再做 ACL 二次校验，最后组装上下文、生成和校验。不要让模型自己决定能看哪些文档。

`traceSafe(candidates)` 表示普通 trace 中只保存当前用户有权查看的候选片段 ID、索引版本、聚合过滤原因和耗时，不保存用户无权查看的原文、标题、路径、可枚举 hash 或可反推出敏感信息的标识。被权限过滤的具体资源只能进入受控安全审计日志。RAG 排障需要可复盘，但复盘日志不能变成新的数据泄露渠道。

### RAG Trace

RAG 系统必须能复盘一次回答，但 trace 默认不应该保存完整原文，也不应该保存用户无权资源的具体标识。建议优先记录当前用户可见材料的 ID、版本、过滤条件和耗时；只有在数据分级、用户授权、留存策略和调试权限都允许时，才保存问题原文或片段正文。至少记录：

- trace_id。
- user_pseudonym、tenant_id、project_id。`user_pseudonym` 应使用带密钥的 HMAC、审计域专用 pseudonym 或内部主体引用，不要用普通 `sha256(user_id)` 伪装成匿名化。
- question_hash，必要且合规时再保存 question 摘要或原文。
- query_hash，必要且合规时再保存 query_text。
- metadata_filter。
- index_version。
- candidate_chunk_ids。
- selected_chunk_ids。
- dropped_chunk_reason。
- answer_citation_ids。
- citation_validation_result。
- latency_ms。
- token_estimate。
- model_requested 和 model_resolved。

敏感场景下不要直接记录用户原文和文档原文，可以记录 hash、chunk_id、脱敏摘要和权限范围。调试全文需要受控权限、短期留存和审计。

## 适用场景

### 玩具 Demo

Demo 阶段可以用几篇 Markdown 文档、简单切分和本地索引跑通流程。目标是理解“问题 -> 检索 -> 注入 -> 生成”的链路。

这个阶段不需要复杂权限和灰度，但要从第一天保留 `source` 和 `chunk_id`。否则 Demo 很容易养成“答案没有来源”的坏习惯。

### 个人效率工具

个人工具适合用 RAG 管理笔记、学习资料、代码片段、读书摘录和个人文档。

重点是：

- 能否快速导入和更新资料。
- 能否看到答案引用。
- 能否删除或重建索引。
- 是否避免把私人资料发送到不合适的模型或服务。

个人场景权限简单，但隐私很重要。

### 团队内部工具

团队知识库、研发问答、客服手册、运维手册和项目文档都适合 RAG。

团队场景开始出现：

- 项目权限。
- 文档版本。
- 多人编辑。
- 过期文档。
- 审计日志。
- 评估集。

此时不能只看“回答像不像”，而要看“引用是否正确、权限是否正确、版本是否正确”。

### 企业级系统

企业级 RAG 面向多租户、多权限、多数据源和合规要求。

它需要处理：

- 租户隔离。
- 字段级或文档级权限。
- 数据分级和脱敏。
- 索引版本和回滚。
- 多语言和多区域数据。
- 缓存、限流和成本控制。
- 高风险答案的人工确认。

企业级 RAG 不是一个向量库加聊天框，而是一条受治理的数据检索和生成链路。

## 不适用场景

不适合把 RAG 用在规则明确、可直接用代码解决的问题。例如金额计算、权限判定、库存扣减、状态机流转。这些应该由确定性程序处理。

不适合用 RAG 替代权限系统。模型不应该看到用户无权访问的材料。

不适合用 RAG 解决高频变化的实时状态，除非索引刷新和数据一致性有保障。例如秒级库存、支付状态和实时风控信号通常应该通过工具或数据库查询，而不是等待文档索引更新。

不适合在没有文档质量治理的情况下上生产。脏文档、重复文档、过期文档和冲突文档会让 RAG 生成看似有来源但实际错误的答案。

不适合把 RAG 当成消除幻觉的万能药。RAG 可以降低缺知识导致的幻觉，但检索错、注入错、引用错时，模型仍然会错。

## 常见坑与反模式

第一个坑是“有向量库就等于有 RAG”。向量库只是检索基础设施，RAG 还需要文档治理、权限、上下文注入、引用和评估。

第二个坑是“切分越小越精准”。片段太小会丢失条件、例外和上下文，模型可能基于半句话回答。

第三个坑是“top-k 越大越好”。召回更多片段会增加成本和噪声，也可能把相互冲突的材料塞给模型。

第四个坑是“只在检索后再过滤权限”。召回后 ACL 二次校验是必要防线，但不能替代查询阶段的租户、项目、数据分级和可见性过滤。如果无权内容已经进入候选上下文、重排输入或普通日志，安全边界可能已经被破坏。权限应贯穿查询构造、索引过滤、召回后校验和上下文注入。

第五个坑是“答案有引用就可信”。引用可能不支持结论，也可能来自过期文档。引用需要校验。

第六个坑是“只评估最终回答”。RAG 还要评估召回、排序、权限过滤、引用和上下文压缩。

第七个坑是“索引建完就不管”。文档会更新、删除、变更权限，索引也需要刷新、失效和回滚。

## 安全、成本与性能考虑

安全方面，RAG 最大风险是越权检索、间接 Prompt Injection 和不可信文件入库。外部文档中的文本不能被当成系统指令，上传文件和网页内容也不能被当作天然安全材料。检索片段应标注为不可信材料，并由工具权限、后端校验和沙箱控制实际动作。

权限方面，要先确定用户可见范围，再构造检索条件。索引记录必须带租户、项目、可见性、数据分级等元数据。召回后还要二次校验，避免索引脏数据或权限漂移。

成本方面，RAG 会增加检索、重排、上下文 token 和引用校验成本。常见控制手段包括：清理重复文档、限制候选数量、裁剪片段、缓存稳定查询、复用索引、对高成本重排做按需触发。

性能方面，在线链路可能慢在查询改写、向量检索、关键词检索、重排模型、权限服务、模型生成和引用校验。要记录分段耗时，而不是只看模型响应时间。

可靠性方面，检索失败要可降级。如果知识库不可用，系统可以回答“当前无法访问知识库”，但不能编造内部文档结论。如果权限服务不可用，高风险场景应 fail closed。

治理方面，要管理索引版本、文档版本、权限版本、切分策略版本和 Prompt 版本。RAG 策略变更要能灰度、回滚和跑回归评估集。

## 如何评估效果

RAG 评估要拆开看，不要只看最终答案。

| 层级 | 指标 | 问题 |
| --- | --- | --- |
| 检索 | 关键片段召回率 | 应该被找到的材料是否被找到 |
| 排序 | Top 结果相关性 | 最有用片段是否排在前面 |
| 权限 | 越权过滤正确率 | 无权材料是否被排除 |
| 上下文 | 入选片段精确率 | 放进模型的材料是否真正有用 |
| 生成 | 答案正确性 | 结论是否被材料支持 |
| 引用 | 引用一致性 | 引用是否真的支持答案 |
| 成本 | token 和延迟 | 是否在业务可接受范围内 |
| 复盘 | trace 完整度 | 错误能否定位到链路环节 |

一个评估样本可以包含：

```json
{
  "case_id": "rag_release_001",
  "question": "上线前还缺哪些评估样本？",
  "user_permission": "team-a",
  "available_docs": ["kb-checklist-v20260520", "old-checklist-v20260401"],
  "expected_retrieved_chunks": ["kb-checklist-20260520-003"],
  "forbidden_chunks": ["team-b-private-plan-001"],
  "expected_excluded_chunks": ["old-checklist-20260401", "team-b-private-plan-001"],
  "expected_dropped_reason": {
    "old-checklist-20260401": "outdated_version",
    "team-b-private-plan-001": "permission_denied"
  },
  "expected_answer_points": [
    "至少 20 条常见问题评估样本",
    "至少 5 条越权访问评估样本"
  ],
  "expected_citations": ["kb-checklist-20260520-003"],
  "failure_type": null
}
```

常见评估集要覆盖：

- 精确术语问题。
- 语义改写问题。
- 文档版本冲突。
- 用户无权限问题。
- 检索到相似但错误文档。
- 多片段组合回答。
- 文档中包含 Prompt Injection。
- 答案引用不支持结论。

如果答案错了，要先问：是文档没入库、切分错、索引错、检索错、排序错、权限错、上下文注入错，还是模型生成错？不同原因对应不同修复动作。

## 实践任务

下面是一组练习材料：

```text
文档 A：上线检查清单
权限：team-a
版本：2026-05-20
内容：上线前必须完成权限过滤测试、安全评审、20 条常见问题评估样本和 5 条越权访问评估样本。若安全评审未通过，只允许内部测试组使用。

文档 B：旧版上线说明
权限：team-a
版本：2026-04-01
内容：上线前准备 10 条常见问题样本即可。

文档 C：B 项目发布计划
权限：team-b
版本：2026-05-25
内容：B 项目计划 2026-06-01 上线。
```

1. 最小任务：设计一个 RAG 上下文包，回答“知识库问答助手上线前还缺哪些评估样本？”交付物包含入选片段、引用 ID、版本和回答草稿。自查标准是：答案必须引用文档 A，不能使用文档 B 的旧规则。

2. 工程化任务：设计文档入库结构。交付物包含 `document_id`、`chunk_id`、`content`、`version`、`tenant_id`、`project_id`、`visibility`、`updated_at`。自查标准是：能解释这些字段如何支持权限过滤、版本判断和引用。

3. 权限任务：模拟 team-a 用户和 team-b 用户分别提问。交付物包含检索过滤条件、被召回片段、被丢弃片段和 `dropped_reason`。自查标准是：team-a 用户不能看到文档 C，team-b 用户不能看到文档 A。

4. 评估任务：构建 8 条 RAG 评估样本，覆盖旧文档、越权文档、语义改写、精确关键词、Prompt Injection、多片段组合、引用错误和知识库不可用。自查标准是：每条样本都能定位错误属于入库、检索、排序、权限、注入、生成还是引用校验。

参考答案要点：回答“上线前还缺哪些评估样本”时，文档 A 是当前有效依据，文档 B 是旧版本，应被丢弃或标为过期，文档 C 对 team-a 用户不可见。team-a 用户不能看到文档 C 的标题、内容或项目计划；team-b 用户不能看到文档 A 的内容。正确答案应引用文档 A，并指出至少 20 条常见问题评估样本和至少 5 条越权访问评估样本。

## 从入门到专业

- 入门：知道 RAG 是“检索外部材料再生成”，不是让模型永久记住知识。
- 初级：能搭建最小文档入库、切分、检索和上下文注入流程。
- 中级：能处理元数据、权限过滤、引用、版本和基础评估。
- 高级：能设计混合检索、重排、缓存、索引刷新、错误归因和回归测试。
- 专业：能构建多租户、可审计、可治理、可回滚的企业级 RAG 平台。

## 本章小结

RAG 解决的是模型不知道外部知识时，如何按需检索材料并让模型基于材料回答。它的核心不是向量数据库，而是从数据源到索引、从检索到上下文、从引用到评估的一整条工程链路。

好的 RAG 系统必须同时处理文档质量、切分策略、检索方式、权限过滤、上下文注入、引用校验、成本性能和评估复盘。只做“Embedding + top-k + Prompt”可以跑 Demo，但很难进入生产。

对 Agent 开发者来说，RAG 是知识入口。第 8 章会继续深入高级 RAG：chunk 策略、Hybrid Search、多路召回、Query Rewrite、Rerank 和效果评估。第 9 章会讨论记忆系统，解决跨会话状态和长期用户偏好的问题。
