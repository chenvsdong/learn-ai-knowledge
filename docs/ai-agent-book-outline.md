# 全书大纲与写作逻辑

这本书的核心目标，不是把 AI 相关名词堆成百科，而是把它整理成一条可以持续学习、持续扩写、持续落地的路径。

我们会按 AI Agent 能力的生长顺序来组织内容：

> 模型如何工作 -> 如何和模型沟通 -> 如何给模型知识 -> 如何让模型调用工具 -> 如何让模型完成任务 -> 如何让 Agent 工程化、产品化、系统化。

## 为什么不用“术语列表”来写

如果只按 LLM、Prompt、RAG、Function Call、MCP、Skill 这些词平铺展开，很容易学成一组孤立概念。你会知道每个词是什么意思，但不一定知道它们如何组合成一个真正能工作的 AI Agent。

更好的方式是按系统能力来组织：

- LLM 是大脑基础。
- Prompt 和 Context 是沟通方式。
- RAG 和 Memory 是知识与记忆。
- Function Call、Tool Use、MCP 和 Skill 是行动能力。
- Planning、Runtime 和 Multi-Agent 是 Agent 架构。
- 评估、安全、性能和成本是生产落地能力。

## 内容持续更新机制

这个 Web Book 采用内容驱动结构：

- 目录由 `content/book.json` 管理。
- 每章正文由 `content/chapters/*.md` 管理。
- 页面代码只负责读取和渲染内容。
- 后续我们通过对话扩写章节时，只需要新增或修改对应 Markdown 文件。
- 部署后，页面会自动展示最新章节内容。

这意味着它不是一次性写完的文档，而是一个可以不断生长的 AI Agent 知识系统。

## 全书结构

这一部分解决一个核心问题：我们到底在和什么东西打交道？

- 主流模型与能力边界

### 第二部分：和模型沟通：Prompt 与上下文工程

这一部分解决一个核心问题：如何让模型稳定听懂你？

- 结构化输出与可靠性

### 第三部分：给模型知识：RAG 与记忆系统

- RAG 进阶
- Agent 的记忆系统

## 每章固定写法

每一章都会尽量保持三层结构：

1. 基础解释：这是什么，为什么需要它。
2. 工程使用：怎么在真实项目里使用它。
3. 深入理解：常见坑、设计权衡、性能、安全和评估问题。

这套结构可以避免两种常见问题：只讲概念但不会落地，或者只堆代码但缺少理解。

## 后续扩写方式

后续我们可以按两种方式推进：

- 顺序写作：从第 1 章开始，一章一章扩写。
- 主题写作：先挑当前最需要的主题，比如 RAG、MCP、Agent Runtime、Java 工程化。

无论采用哪种方式，最终都会沉淀到同一套 Web Book 结构中。
