# 章节写作规范：AI Agent 专业化写书协议

这份规范用于约束《AI Agent 开发：从大模型原理到工程化落地》的每一章写作。目标不是让 AI 自由生成一篇文章，而是让每一章都经过设计、写作、审查和修订，最终能指导读者逐步成长为专业的 AI Agent 开发工程师。

## 核心原则

每一章都必须回答一条完整链路：

> 是什么 -> 为什么 -> 怎么做 -> 怎么做好 -> 什么时候别这么做 -> 如何验证自己做对了。

写作时必须同时满足四个标准：

- 概念正确：术语、机制、边界不能含糊或编造。
- 工程可落地：能映射到真实系统、接口、数据流、异常处理和运维问题。
- 场景有深度：不只讲 Demo，还要讲个人工具、团队工具、企业系统中的差异。
- 学习可推进：读者能从入门理解逐步走向专业实践。

## 强制工作流

写任何章节时，不允许直接进入正文。必须按以下步骤推进：

1. 定义本章目标
   - 读者读完本章应该理解什么？
   - 读者应该能够做什么？
   - 本章和前后章节有什么关系？

2. 列出知识点地图
   - 基础概念
   - 核心原理
   - 工程实现
   - 适用场景
   - 不适用场景
   - 常见坑与反模式
   - 安全、成本、性能、评估问题
   - 实践任务

3. 写第一版正文
   - 先保证结构完整。
   - 不为了流畅牺牲准确性。
   - 不把未经核查的内容写成确定事实。

4. 做事实核查
   - 涉及官方协议、API、SDK、框架版本、命令、配置时，必须核查官方文档或本地项目。
   - 涉及行业趋势或模型能力时，必须标注时间背景。
   - 不写无来源的精确性能数字。

5. 做完整性检查
   - 是否讲清楚问题？
   - 是否讲清楚原理？
   - 是否讲清楚工程实现？
   - 是否覆盖适用和不适用场景？
   - 是否覆盖常见坑？
   - 是否覆盖安全、成本、性能和评估？
   - 是否有实践任务？

6. 做工程化检查
   - 是否能映射到真实后端系统？
   - 是否有输入、处理、输出、状态、异常的说明？
   - 是否说明如何观测、评估、调试？
   - 是否避免了只能跑 Demo、不能进生产的写法？

7. 做学习体验检查
   - 初学者是否能读懂？
   - 有经验的工程师是否有收获？
   - 是否由浅入深？
   - 是否有足够例子帮助建立直觉？

8. 修订章节文件
   - 如果审查发现问题，必须直接修订正文。
   - 不只在审查里指出问题而不改。

## 每章固定结构

除非有明确理由，每一章应使用以下结构：

```markdown
# 第 X 章：章节标题

## 本章解决什么问题

## 一个直观例子

## 基础解释

## 核心原理

## 工程实现

## 适用场景

## 不适用场景

## 常见坑与反模式

## 安全、成本与性能考虑

## 如何评估效果

## 实践任务

## 从入门到专业

## 本章小结
```

## 幻觉防护规则

### 内容类型分级

不同内容采用不同严格度：

- 概念解释：可以先由模型生成，但必须检查逻辑和术语边界。
- 技术标准：必须查官方文档，例如 MCP、OpenAI API、Spring AI、LangChain4j。
- 框架 API：必须基于真实版本，不确定时不要写死方法名。
- 行业趋势：必须标注时间背景，例如“截至 2026 年”。
- 性能数字：没有来源时不要给精确数字。
- 最佳实践：应说明是工程经验判断，不伪装成绝对标准。

### 禁止行为

- 禁止编造 API 名称、类名、方法名、配置项。
- 禁止编造框架能力。
- 禁止编造 benchmark、QPS、延迟、成本下降比例。
- 禁止把推测写成确定事实。
- 禁止用“业界通常”“很多公司”这类表达替代事实来源。
- 禁止为了让内容完整而补虚构案例。

### 允许但要标注的内容

- 架构草图可以使用概念模型，但要说明是抽象模型。
- 代码可以使用伪代码，但要明确标注。
- 趋势判断可以写，但要说明是基于当前观察的判断。
- 工程建议可以写，但要说明适用条件和边界。

## 内容完整性检查清单

每章写完后必须检查：

- 这一章有没有讲清楚它解决的问题？
- 有没有给出最小可理解模型？
- 有没有解释核心原理？
- 有没有工程架构视角？
- 有没有真实使用场景？
- 有没有不适用场景？
- 有没有常见坑与反模式？
- 有没有安全、成本、性能、评估角度？
- 有没有实践任务？
- 有没有从入门到专业的能力推进？
- 有没有和前后章节建立连接？

## 场景深度分层

每个关键技术点都尽量按四个层级展开：

1. 玩具 Demo
   - 目标是跑通概念。
   - 适合理解流程，不代表生产能力。

2. 个人效率工具
   - 解决个人真实问题。
   - 开始考虑体验、数据管理和长期使用。

3. 团队内部工具
   - 多人使用。
   - 必须考虑权限、日志、稳定性、数据隔离。

4. 企业级系统
   - 面向生产。
   - 必须考虑审计、安全、成本、可观测性、SLA、回滚和治理。

## 专业化推进

每章都要说明读者如何从入门走向专业。例如：

- 入门：知道概念是什么，能解释基本流程。
- 初级：能跑通一个最小 Demo。
- 中级：能把它接入真实后端系统。
- 高级：能处理异常、权限、评估、成本和性能问题。
- 专业：能在完整 Agent 平台中设计、治理和优化这项能力。

## 审稿角色

写作过程中必须切换以下角色：

1. 作者
   - 负责完成结构完整的初稿。

2. 技术审稿人
   - 检查概念错误、幻觉风险、术语不准、事实不确定。

3. 工程审稿人
   - 检查是否能落地到真实后端系统。
   - 检查是否覆盖状态、异常、权限、日志、评估、部署问题。

4. 学习体验审稿人
   - 检查是否由浅入深。
   - 检查是否对初学者友好。
   - 检查是否给有经验的工程师足够深度。

5. 主编
   - 统一章节结构、语气、节奏和前后章节连接。

```text
   - 本章解决什么问题

   - 核心原理

   - 适用场景
   - 不适用场景

```
