# 章节写作规范：AI Agent 专业化写书协议

这份规范用于约束《AI Agent 开发：从大模型原理到工程化落地》的每一章写作。目标不是让 AI 自由生成一篇文章，而是让每一章都经过设计、写作、审查和修订，最终能指导读者逐步成长为专业的 AI Agent 开发工程师。

## 核心原则

每一章都必须回答一条完整链路：

> 是什么 -> 为什么 -> 怎么做 -> 怎么做好 -> 什么时候别这么做 -> 如何验证自己做对了。

写作时必须同时满足四个标准：

- 概念正确：术语、机制、边界不能含糊或编造。
- 工程可落地：能映射到真实系统、接口、数据流、异常处理和运维问题。
- 场景有深度：不只讲 Demo，还要讲个人工具、团队工具、企业系统中的差异。
- 学习可推进：读者能从入门理解逐步走向专业实践。

## 强制工作流

写任何章节时，不允许直接进入正文。必须按以下步骤推进：

1. 章节架构师定义本章目标
   - 读者读完本章应该理解什么？
   - 读者应该能够做什么？
   - 本章和前后章节有什么关系？

2. 章节架构师列出知识点地图
   - 基础概念
   - 核心原理
   - 工程实现
   - 适用场景
   - 不适用场景
   - 常见坑与反模式
   - 安全、成本、性能、评估问题
   - 实践任务

3. 作者写第一版正文
   - 先保证结构完整。
   - 不为了流畅牺牲准确性。
   - 不把未经核查的内容写成确定事实。

4. 技术审稿人做事实核查
   - 涉及官方协议、API、SDK、框架版本、命令、配置时，必须核查官方文档或本地项目。
   - 涉及行业趋势或模型能力时，必须标注时间背景。
   - 不写无来源的精确性能数字。

5. 技术审稿人做概念完整性检查
   - 是否讲清楚问题？
   - 是否讲清楚原理？
   - 是否覆盖适用和不适用场景？
   - 是否覆盖常见坑？

6. 工程审稿人做工程化检查
   - 是否能映射到真实后端系统？
   - 是否有输入、处理、输出、状态、异常的说明？
   - 是否说明如何观测、评估、调试？
   - 是否避免了只能跑 Demo、不能进生产的写法？
   - 是否覆盖安全、成本、性能和评估？

7. 学习体验审稿人做学习体验检查
   - 初学者是否能读懂？
   - 有经验的工程师是否有收获？
   - 是否由浅入深？
   - 是否有足够例子帮助建立直觉？
   - 是否有实践任务？

8. 主编汇总审稿意见并决定修订方向
   - 哪些意见必须立即修订？
   - 哪些内容应移动到后续章节？
   - 哪些建议不采纳，原因是什么？

9. 作者修订章节文件
   - 如果审查发现问题，必须直接修订正文。
   - 不只在审查里指出问题而不改。

10. 主编做最终一致性检查
    - 统一章节结构、语气和节奏。
    - 检查和前后章节的衔接。
    - 更新简洁的写作审查记录。

## 每章固定结构

除非有明确理由，每一章应使用以下结构：

```markdown
# 第 X 章：章节标题

## 本章解决什么问题

## 一个直观例子

## 基础解释

## 核心原理

## 工程实现

## 适用场景

## 不适用场景

## 常见坑与反模式

## 安全、成本与性能考虑

## 如何评估效果

## 实践任务

## 从入门到专业

## 本章小结
```

## 幻觉防护规则

### 内容类型分级

不同内容采用不同严格度：

- 概念解释：可以先由模型生成，但必须检查逻辑和术语边界。
- 技术标准：必须查官方文档，例如 MCP、OpenAI API、Spring AI、LangChain4j。
- 框架 API：必须基于真实版本，不确定时不要写死方法名。
- 行业趋势：必须标注时间背景，例如“截至 2026 年”。
- 性能数字：没有来源时不要给精确数字。
- 最佳实践：应说明是工程经验判断，不伪装成绝对标准。

### 禁止行为

- 禁止编造 API 名称、类名、方法名、配置项。
- 禁止编造框架能力。
- 禁止编造 benchmark、QPS、延迟、成本下降比例。
- 禁止把推测写成确定事实。
- 禁止用“业界通常”“很多公司”这类表达替代事实来源。
- 禁止为了让内容完整而补虚构案例。

### 允许但要标注的内容

- 架构草图可以使用概念模型，但要说明是抽象模型。
- 代码可以使用伪代码，但要明确标注。
- 趋势判断可以写，但要说明是基于当前观察的判断。
- 工程建议可以写，但要说明适用条件和边界。

## 内容完整性检查清单

每章写完后必须检查：

- 这一章有没有讲清楚它解决的问题？
- 有没有给出最小可理解模型？
- 有没有解释核心原理？
- 有没有工程架构视角？
- 有没有真实使用场景？
- 有没有不适用场景？
- 有没有常见坑与反模式？
- 有没有安全、成本、性能、评估角度？
- 有没有实践任务？
- 有没有从入门到专业的能力推进？
- 有没有和前后章节建立连接？

## 场景深度分层

每个关键技术点都尽量按四个层级展开：

1. 玩具 Demo
   - 目标是跑通概念。
   - 适合理解流程，不代表生产能力。

2. 个人效率工具
   - 解决个人真实问题。
   - 开始考虑体验、数据管理和长期使用。

3. 团队内部工具
   - 多人使用。
   - 必须考虑权限、日志、稳定性、数据隔离。

4. 企业级系统
   - 面向生产。
   - 必须考虑审计、安全、成本、可观测性、SLA、回滚和治理。

## 专业化推进

每章都要说明读者如何从入门走向专业。例如：

- 入门：知道概念是什么，能解释基本流程。
- 初级：能跑通一个最小 Demo。
- 中级：能把它接入真实后端系统。
- 高级：能处理异常、权限、评估、成本和性能问题。
- 专业：能在完整 Agent 平台中设计、治理和优化这项能力。

## 审稿角色

写作过程中必须切换以下角色。主会话负责章节架构师、作者和主编；技术审稿人、工程审稿人、学习体验审稿人优先使用上下文干净的子 Agent 或等价的隔离审稿方式执行。隔离审稿的目标是减少作者视角带来的自我确认。

1. 章节架构师
   - 负责正文之前的章节设计。
   - 输出本章目标、知识点地图、边界和前后章节关系。
   - 不直接写正文。

2. 作者
   - 负责完成结构完整的初稿。
   - 按固定章节结构写作。
   - 遇到事实不确定、版本不确定、趋势判断时必须标注边界。

3. 技术审稿人
   - 检查概念错误、幻觉风险、术语不准、事实不确定。
   - 检查是否混淆 AI、机器学习、深度学习、LLM、Agent 等概念。
   - 涉及官方协议、API、SDK、框架版本、命令、配置时，要求核查官方文档或本地项目。
   - 不评价文风，除非文风导致技术含义错误。

4. 工程审稿人
   - 检查是否能落地到真实后端系统。
   - 检查是否覆盖状态、异常、权限、日志、评估、部署问题。
   - 检查个人工具、团队内部工具、企业级系统之间的差异。
   - 检查是否只停留在 Demo，是否缺少生产系统边界。

5. 学习体验审稿人
   - 检查是否由浅入深。
   - 检查是否对初学者友好。
   - 检查是否给有经验的工程师足够深度。
   - 检查例子、实践任务和小结是否能推动读者继续学习。

6. 主编
   - 统一章节结构、语气、节奏和前后章节连接。
   - 汇总审稿意见并决定修订优先级。
   - 确保审稿意见已经进入正文修订，而不是只留在审查记录里。

### 子 Agent 使用规则

当环境提供子 Agent 或多 Agent 工具时，技术审稿人、工程审稿人、学习体验审稿人应优先由独立子 Agent 执行。每个子 Agent 只接收完成任务所需的最小上下文：

- 本章标题。
- 本章目标和知识点地图。
- 当前章节正文。
- 本书写作规范中与该角色相关的部分。
- 该审稿角色的职责和输出格式。

子 Agent 不负责重写整章，不负责统一文风，不负责最终取舍。它只输出：

- 发现的问题。
- 风险等级或影响范围。
- 建议修订动作。
- 是否必须修改正文。

如果当前环境没有可用的子 Agent 工具，也必须在主会话中模拟隔离审稿：每个审稿角色单独执行，不共享前一个审稿角色的结论作为前提，最后再由主编统一汇总。

### 审查记录要求

每章末尾必须保留简洁的 `写作审查记录`。审查记录不是形式化签名，必须说明：

- 各角色发现的核心问题。
- 已经采取的修订动作。
- 哪些问题留到后续章节展开。

审稿发现的问题必须优先修订正文。禁止只在审查记录中承认问题，却不修改正文。

## 后续对话中的推荐指令

扩写章节时，推荐使用类似指令：

```text
请按本书写作规范扩写第 X 章。

要求：
1. 先输出本章目标和知识点地图，不要直接写正文。
2. 我确认结构后，再写正文。
3. 正文必须包含：
   - 本章解决什么问题
   - 一个直观例子
   - 基础解释
   - 核心原理
   - 工程实现
   - 适用场景
   - 不适用场景
   - 常见坑与反模式
   - 安全、成本与性能考虑
   - 如何评估效果
   - 实践任务
   - 从入门到专业
   - 本章小结
4. 写完后，以技术审稿人、工程审稿人、学习体验审稿人、主编四个角色检查。
5. 如果发现问题，直接修订章节文件。
```
